# 스레드 (Thread)

> <b>💡 한 줄 요약</b>
> 스레드는 프로세스 안에서 <b>메모리(code, data, OS resource)는 공유</b>하면서 <b>실행 상태(PC, register, stack)만 따로</b> 갖는 CPU 실행 단위입니다. 프로세스보다 가볍게 만들고 전환할 수 있어 <b>응답성과 효율</b>을 동시에 높입니다.

---

## 1. 스레드(Thread)란?

> **스레드 = CPU가 일을 처리하는 가장 기본 단위** (basic unit of CPU utilization)

하나의 프로세스 안에서 여러 스레드가 같은 메모리 공간을 공유하면서, 각자 자기만의 실행 상태를 갖고 동시에 실행됩니다.

### 개인 소유 vs 공유

| 구분 | 항목 | 설명 |
| --- | --- | --- |
| **개인 소유** | Thread ID | 스레드를 구분하는 식별자 |
| | Program Counter (PC) | 지금 코드의 어디를 실행 중인지 위치 표시 |
| | Register set | 지금 계산 중인 값들을 담는 임시 저장 공간 |
| | Stack | 함수 호출 기록, 지역 변수 — 각자 하는 일이 다르니 따로 필요 |
| **공유** | Code section | 프로그램 코드 |
| | Data section | 전역 변수 |
| | OS resources | 열어놓은 파일 등 |

> 💡 이 공유되는 묶음을 **task**라고 부릅니다. 스레드가 딱 하나만 있는 전통적인 프로세스는 **heavyweight process**, 스레드는 그보다 가볍다는 의미에서 **lightweight process(경량 프로세스)**라고도 부릅니다.

---

## 2. 왜 멀티스레드를 쓰는가?

핵심 상황: **한 스레드가 멈춰서 기다리는 동안(blocked/waiting), 같은 task 안의 다른 스레드가 계속 일할 수 있다.**

| 상황 | 프로세스 버전 | 스레드 버전 |
| --- | --- | --- |
| Producer-Consumer 문제, 버퍼가 꽉 참 | 생산자 프로세스가 **통째로 block** | 생산자 스레드만 멈추고 **소비자 스레드는 계속 진행** |

### 장점 4가지 — 시험 단골

| 장점 | 설명 |
| --- | --- |
| **Responsiveness** (응답성) | 하나가 막혀도 다른 스레드가 일함 |
| **Resource Sharing** (자원 공유) | code, data, 자원을 공유 |
| **Economy** (경제성) | 공유 덕분에 생성/전환 비용이 프로세스보다 훨씬 저렴 |
| **Utilization of MP Architectures** (멀티프로세서 활용) | 여러 CPU 코어에 여러 스레드를 나눠 돌릴 수 있음 |

## 3. 멀티프로세스 vs 멀티스레드

| | 멀티프로세스 | 멀티스레드 |
| --- | --- | --- |
| 독립성 | 서로 **완전히 독립** | 서로 독립적이지 **않음** (한 프로세스에서 파생) |
| 메모리 | **별도의 주소 공간** 사용 | **같은 주소 공간 공유** → context switch 시 메모리 관리 오버헤드 없음 |
| 소유자 | 서로 다른 사용자의 프로세스일 수 있음 → **상호 보호 필요** | 하나의 task는 **한 사용자만** 소유 → 상호 보호 불필요 |
| 구성 | PC + register set + stack **pointer** | PC + register set + stack **space** |

> **용어 — context switch(문맥 교환):** CPU가 지금 실행 중인 작업을 멈추고 다른 작업으로 갈아타는 것. 현재 작업의 상태(레지스터, PC 등)를 저장하고 다음 작업의 상태를 불러와야 합니다. 프로세스 간 전환은 메모리 공간까지 통째로 갈아타야 해서 비싸고, 스레드 간 전환은 메모리를 공유하니 훨씬 쌉니다.

### 장단점 요약

| 장점 | 단점 |
| :--- | :--- |
| **전환 비용이 낮음**<br>주소 공간을 공유하므로 context switch 시 메모리 관리 오버헤드가 없어 프로세스보다 훨씬 빠르게 전환됨 | **상호 보호가 안 됨**<br>한 스레드의 오류(잘못된 메모리 접근 등)가 같은 task 안의 다른 모든 스레드, 즉 프로세스 전체에 영향을 줄 수 있음 |

## 4. User Level Thread vs Kernel Level Thread

스레드를 "누가 만들고 관리하느냐"에 따른 구분입니다.

- **Kernel level thread**: OS 커널이 직접 생성·관리. **CPU(커널)가 인식하는 건 이것뿐**
- **User level thread**: 사용자 공간에서 라이브러리로 관리. 커널은 그 존재를 모름 → 커널 레벨 스레드를 통해서만 CPU에 작업이 전달됨

| | User level threads | Kernel level threads |
| --- | --- | --- |
| 전환 방식 | 커널 개입 없음 | 커널이 (인터럽트를 통해) 전환을 수행 |
| 전환 속도 | **빠름** — system call → 커널 인터럽트 오버헤드가 없기 때문 | 상대적으로 **느림** |
| 스케줄링 공평성 | 불공평 (커널은 프로세스 단위로만 스케줄) | 스레드마다 독립적으로 스케줄 가능 |

> ⭐ 핵심 문장: **"CPU는 커널 레벨 스레드만 본다. 유저 레벨 스레드는 CPU에게 보이지 않는다."**

<details>
<summary>💡 "불공평한 스케줄링"이란? — P1/P2 예시</summary>

프로세스 P1에 유저 스레드 10개, P2에 2개가 있다고 합시다 (총 12개).

- **유저 레벨 스레드만 지원되는 경우**: CPU(커널)는 스레드가 보이지 않고 **프로세스 2개만** 보입니다. 그래서 P1과 P2에 CPU 시간을 반반씩 줍니다 → P1의 스레드 10개는 절반의 시간을 10등분해서 나눠 쓰고, P2의 스레드 2개는 절반을 2등분해서 씁니다. **스레드 입장에서 불공평**하죠.
- **커널 레벨 스레드도 지원되는 경우**: 커널이 12개 스레드를 전부 인식하고 각각 독립적으로 스케줄할 수 있습니다.

</details>

### 구현 예시

| 구분 | 예시 |
| --- | --- |
| 커널 지원 스레드 | Mach, OS/2 |
| 커널 스레드를 쓰는 OS | Windows XP/2000, Solaris, Linux, Mac OS X |
| 유저 레벨 스레드 라이브러리 3대장 | POSIX Pthreads, Win32 threads, Java threads (CMU의 Project Andrew에서 유래) |
| Hybrid (둘 다 지원) | Solaris 2 |

## 5. 멀티스레딩 모델 — 유저 스레드와 커널 스레드의 매핑

유저 레벨 스레드가 커널 레벨 스레드에 **어떻게 연결(매핑)되느냐**의 문제입니다.

| 모델 | 매핑 | 장점 | 단점 | 예시 |
| --- | --- | --- | --- | --- |
| **① Many-to-One** | 여러 유저 스레드 → 커널 스레드 **1개** | 유저 공간에서 관리, 생성 비용 저렴 | 커널 스레드 하나가 block되면 **유저 스레드 전부 block** | Solaris Green Threads, GNU Portable Threads |
| **② One-to-One** | 유저 스레드 1개 ↔ 커널 스레드 1개 | 한 스레드가 block되어도 **다른 스레드는 계속 실행** | 유저 스레드를 만들 때마다 커널 스레드도 생성해야 함 (비용) | Windows NT/XP/2000, Linux, Solaris 9 이후 |
| **③ Many-to-Many** | 여러 유저 스레드 → **여러** 커널 스레드 (M:N) | OS가 커널 스레드를 충분한 수만큼 만들어 유연하게 배분 | 구현이 복잡 | Solaris 9 이전, Windows NT/2000 + ThreadFiber |
| **④ Two-level** | M:M 기반 + 중요한 유저 스레드는 커널 스레드에 **1:1 고정(bound)** 가능 | 유연성 + 필요한 곳만 안정성 확보 | 구현이 가장 복잡 | IRIX, HP-UX, Tru64 UNIX, Solaris 8 이전 |

## 6. Threading Issues (스레드 관련 쟁점들)

### 6-1. fork()와 exec()의 의미론

질문: 멀티스레드 프로세스에서 한 스레드가 `fork()`를 부르면, **호출한 스레드만 복제? 아니면 스레드 전부 복제?**

| 상황 | 동작 |
| --- | --- |
| `fork()` 직후 바로 `exec()`을 부를 경우 | **호출한 스레드만** 복제 (어차피 exec()이 전부 덮어쓰므로 다 복제하면 낭비) |
| `exec()`을 바로 안 부를 경우 | **모든 스레드** 복제 |

### 6-2. Thread Cancellation (스레드 취소)

스레드가 일을 끝내기 전에 강제 종료시키는 것.

| 방식 | 설명 |
| --- | --- |
| **Asynchronous cancellation** (즉시 취소) | 대상 스레드를 바로 종료 |
| **Deferred cancellation** (지연 취소) | 대상 스레드가 주기적으로 "나 취소해야 하나?" 스스로 확인 후 종료 → **자원을 정리하고 죽을 수 있어 더 안전** |

### 6-3. Signal Handling (시그널 처리)

시그널 = UNIX에서 프로세스에게 "특정 이벤트가 발생했다"고 알리는 방법.

처리 흐름: ① 이벤트로 시그널 **생성(generated)** → ② 프로세스에 **전달(delivered)** → ③ **처리(handled)**

멀티스레드에서는 "누구에게 전달할까?"가 문제입니다. 4가지 옵션:

1. 시그널이 해당되는 스레드에게만
2. 프로세스 내 **모든** 스레드에게
3. 프로세스 내 **특정** 스레드들에게
4. 시그널 전담 스레드 하나를 지정해서 전부 그쪽으로

### 6-4. Thread Pools (스레드 풀)

스레드를 **미리 여러 개 만들어 풀(pool)에 대기**시켜 두고, 작업이 오면 꺼내 쓰고 끝나면 반납.

| 장점 | 설명 |
| --- | --- |
| 속도 | 요청마다 새 스레드를 만드는 것보다 빠름 |
| 안정성 | 스레드 개수가 풀 크기로 제한됨 → 무한정 늘어나 시스템이 터지는 걸 방지 |

### 6-5. Thread Specific Data (스레드 고유 데이터)

스레드마다 **자기만의 데이터 사본**을 갖게 하는 기능.

- 스레드 생성을 직접 통제할 수 없을 때(예: 스레드 풀 사용 시) 유용
- 예: 은행 시스템에서 트랜잭션 하나 = 스레드 하나 → 각 거래 정보는 섞이면 안 됨

### 6-6. Scheduler Activations (스케줄러 활성화)

M:M / Two-level 모델에서는 "커널 스레드를 몇 개 유지할지" 커널과 유저 라이브러리가 **소통**해야 합니다.

- **LWP (Lightweight Process)**: 유저 스레드 라이브러리 입장에서의 **가상 프로세서**. 유저 스레드와 커널 스레드 사이의 중간 레이어
- 구조: 유저 스레드 ↔ LWP (M:M) ↔ 커널 스레드 (LWP:커널스레드 = 1:1)
- **Upcall**: 커널이 스레드 라이브러리에게 알림을 보내는 통신 메커니즘 → 애플리케이션이 상황(CPU-bound vs IO-bound)에 맞게 커널 스레드 수를 조절할 수 있게 해줌

## 7. OS별 스레드 구현

| 환경 | 특징 |
| --- | --- |
| **Pthreads** | POSIX 표준(IEEE 1003.1c) API. "동작"만 명세하고 구현은 라이브러리 자유. UNIX 계열(Solaris, Linux, Mac OS X)에서 널리 사용 |
| **Windows XP** | **one-to-one** 매핑. 스레드 = thread id + register set + 유저/커널 **분리된 스택** + private 저장 공간. register set·stack·private 영역을 묶어 **context**라 부름. 자료구조: ETHREAD, KTHREAD, TEB |
| **Linux** | 스레드라 안 부르고 **task**라 부름. `clone()` 시스템 콜로 생성 — 자식 task가 부모의 **주소 공간을 공유**할 수 있게 해줌 |
| **Java** | JVM이 관리. 생성 방법: ① Thread 클래스 상속(extends) ② Runnable 인터페이스 구현(implements) |

#### Java Thread States (상태 전이)

```
new --start()--> runnable --(sleep()/I/O)--> blocked
                 runnable <--(I/O 완료)------ blocked
                 runnable --run() 종료--> dead
```

## 💡 핵심 정리

| 구분 | 내용 |
| --- | --- |
| 정의 | 스레드 = CPU 사용의 기본 단위. 개인 소유: ID, PC, registers, stack / 공유: code, data, OS 자원 |
| 장점 | 응답성, 자원 공유, 경제성, 멀티프로세서 활용 |
| 프로세스 vs 스레드 | 프로세스는 독립적(별도 주소 공간, 상호 보호 필요) / 스레드는 주소 공간 공유(전환 비용 낮음, 보호 불필요) |
| User vs Kernel | 유저 스레드는 커널을 안 거쳐 전환이 빠르지만, **CPU는 커널 스레드만 볼 수 있음** |
| 매핑 모델 | Many-to-One(하나 막히면 전부 정지) / One-to-One(안정적이지만 비용) / Many-to-Many(유연) / Two-level(M:M + 중요한 건 1:1 고정) |
| fork() | exec()이 바로 따라오면 호출 스레드만, 아니면 전부 복제 |
| 취소 방식 | deferred가 더 안전 (자원 정리 후 종료) |
| 시그널 전달 | 4가지 옵션 (해당 스레드만 / 전체 / 특정 스레드들 / 전담 스레드) |
| 스레드 풀 | 속도 향상 + 스레드 개수 제한 |
