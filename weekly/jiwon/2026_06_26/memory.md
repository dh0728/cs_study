## 메모리 계층

저장장치는 **속도, 비용, 휘발성**을 기준으로 계층적으로 분류된다. 메인 메모리를 기준으로 위쪽은 빠르지만 휘발성, 아래쪽은 느리지만 비휘발성이다.

```
        빠름 / 비쌈 / 용량 작음
          ▲
   ┌──────────────┐
   │  레지스터     │  CPU 바로 옆, 가장 빠름 (KB)
   ├──────────────┤
   │  캐시(Cache)  │  1차는 CPU 안, 2차는 CPU 밖 (MB)
   ├──────────────┤
   │  메인 메모리   │  RAM (GB)
   ├──────────────┤
   │  디스크 저장소 │  HDD/SSD
   ├──────────────┤
   │  CD / 테이프   │  보조 백업용
   └──────────────┘
          ▼
        느림 / 쌈 / 용량 큼
```

- **메인 메모리 위쪽(레지스터, 캐시):** 휘발성. 정전 시 정보가 사라짐
- **메인 메모리 아래쪽(디스크, 테이프):** 비휘발성. 정전에도 정보가 남음
- CPU에 가까울수록 빠르고, 멀어질수록 용량이 커진다

> 💡 CPU가 원하는 데이터가 가능한 한 메모리 위쪽 계층에서 해결될수록 성능이 좋아진다. 디스크까지 내려가는 횟수가 늘면 성능이 떨어진다.

> 🖼️ **이미지 추천:** 메모리 계층 피라미드 (위로 갈수록 빠름·소용량, 아래로 갈수록 느림·대용량을 화살표로 표현).

## 캐싱

**캐싱(Caching)** 은 자주 쓰일 가능성이 높은 데이터를 더 빠른 상위 계층에 미리 올려두는 것이다.

```
CPU가 데이터를 원할 때:
레지스터 확인 → 캐시 확인 → 메인 메모리 확인 → 없으면 디스크 확인
```

디스크에서 데이터를 가져올 때는 필요한 데이터 하나만 가져오는 게 아니라, 해당 **블록(섹터) 전체**를 메모리로 올린다. 이 때문에 같은 데이터가 여러 계층에 동시에 존재하게 된다.

> 💡 **캐시 일관성 문제(Cache Coherency)**
> 멀티 프로세서 환경에서는 같은 데이터가 여러 CPU의 캐시에 각각 존재할 수 있다. 한 CPU가 값을 바꾸면 다른 캐시의 값도 함께 갱신해야 하는데, 이를 캐시 일관성 문제라고 한다. 보통 버스(bus)를 통해 다른 CPU들도 함께 갱신하는 방식으로 해결한다.

## 논리 주소와 물리 주소

| 구분 | 설명 |
| --- | --- |
| **논리 주소(Logical / Virtual Address)** | 프로그래머·CPU 입장에서 보는 주소. CPU가 발생시킴 |
| **물리 주소(Physical Address)** | 실제 메모리 장치가 보는 주소 |

프로그램이 실행되기까지 주소는 다음 순서로 변환된다.

```
심볼릭 주소        →   재배치 가능 코드      →    절대 주소
(Symbolic Addr)      (Relocatable, 논리)        (Absolute, 물리)
변수 이름 등          컴파일된 상대 주소          실제 메모리 주소
```

주소 바인딩(Address Binding)은 세 시점 중 하나에서 일어날 수 있다.

- **컴파일 타임:** 메모리 위치를 미리 알면 절대 코드 생성. 위치가 바뀌면 재컴파일 필요
- **로드 타임:** 재배치 가능한 코드 생성, 바인딩을 로딩까지 미룸
- **실행 타임:** 실행 중에 바인딩 (대부분의 현대 운영체제)

## MMU와 주소 변환

**MMU(Memory Management Unit)** 는 논리 주소를 물리 주소로 변환해주는 하드웨어다.

```
CPU가 만든 논리 주소 (예: 346)
        │
        ▼
   ┌─────────────────────┐
   │   MMU               │
   │   relocation register │  시작 주소 보유 (예: 14000)
   │   14000 + 346       │
   └─────────────────────┘
        │
        ▼
물리 주소 (예: 14346)
```

시작 주소를 담는 **relocation register(base register)** 가 있어야 실행 중 위치를 바꾸는 동적 재배치(dynamic relocation)가 가능하다.

> 🖼️ **이미지 추천:** CPU(논리주소) → MMU(relocation register로 덧셈) → 물리주소(메모리)로 이어지는 변환 그림.

## 스와핑

**스와핑(Swapping)** 은 메모리에 올라온 프로세스를 일시적으로 디스크(backing store)로 내보냈다가 필요할 때 다시 불러오는 것이다.

- **Backing store:** 모든 프로세스의 메모리 이미지를 담을 수 있는 충분히 빠르고 큰 디스크
- **Roll out, roll in:** 우선순위 기반 스케줄링에서, 낮은 우선순위 프로세스를 내보내고 높은 우선순위 프로세스를 올리는 방식

```
메모리                    디스크(backing store)
[프로세스 A] ──roll out──▶ [프로세스 A 저장]
[프로세스 B] ◀──roll in─── [프로세스 B 불러옴]
```

## 연속 할당과 단편화

### 연속 할당 (Contiguous Allocation)

메인 메모리는 보통 두 영역으로 나뉜다.

- **낮은 주소(low memory):** 운영체제가 사용
- **높은 주소(high memory):** 사용자 프로세스가 사용

프로세스가 들어올 때, 충분히 큰 빈 공간(**hole**)을 찾아 할당한다. 빈 공간을 고르는 방식은 세 가지다.

| 방식 | 설명 | 장점 |
| --- | --- | --- |
| **First-fit** | 충분히 큰 첫 번째 hole에 할당 | 속도가 빠름 |
| **Best-fit** | 충분히 큰 것 중 가장 작은 hole에 할당 | 공간 활용도 좋음 |
| **Worst-fit** | 가장 큰 hole에 할당 | — |

### 단편화 (Fragmentation)

| 종류 | 설명 |
| --- | --- |
| **외부 단편화(External)** | 전체 빈 공간은 충분하지만 **흩어져 있어서** 연속된 공간이 부족한 상태 |
| **내부 단편화(Internal)** | 요청보다 **큰 공간을 할당받아** 남는 공간이 생기는 상태 (예: 10K 요청에 12K 할당 → 2K 낭비) |

> 🖼️ **이미지 추천:** 외부 단편화(공간이 조각조각 흩어진 모습)와 내부 단편화(할당 블록 안에 남는 자투리)를 나란히 비교하는 그림.

## 페이징

**페이징(Paging)** 은 메모리를 고정 크기 조각으로 나눠 관리하는 방식이다. 바둑판처럼 일정한 칸으로 나눈다고 생각하면 된다.

- 물리 메모리를 고정 크기 블록인 **프레임(frame)** 으로 나눔
- 논리 메모리를 같은 크기 블록인 **페이지(page)** 로 나눔
- 프로세스의 논리 주소 공간이 **연속적이지 않아도** 물리 메모리의 빈 프레임 어디든 할당 가능

```
논리 메모리              페이지 테이블          물리 메모리
page 0  ──────────────▶  [0] → 1   ──────────▶  frame 1: page 0
page 1  ──────────────▶  [1] → 4   ──────────▶  frame 4: page 1
page 2  ──────────────▶  [2] → 3   ──────────▶  frame 3: page 2
page 3  ──────────────▶  [3] → 7   ──────────▶  frame 7: page 3
```

페이징 덕분에 외부 단편화는 사라지지만, 페이지 크기보다 작은 자투리로 인한 **내부 단편화**는 여전히 존재한다.

> 🖼️ **이미지 추천:** 논리 메모리의 페이지들이 페이지 테이블을 거쳐 물리 메모리의 흩어진 프레임으로 매핑되는 그림.

### 주소 변환

논리 주소는 두 부분으로 나뉜다.

| 구성 | 의미 |
| --- | --- |
| **페이지 번호(p)** | 페이지 테이블에서 찾기 위한 인덱스 |
| **페이지 오프셋(d)** | 프레임 시작 주소에 더해질 페이지 내 위치 |

```
논리 주소 = (페이지 번호 p, 오프셋 d)
   → 페이지 테이블에서 p로 프레임 번호 f를 찾음
   → 물리 주소 = (프레임 f의 시작 주소 + 오프셋 d)
```

## 페이지 테이블과 TLB

페이지 테이블은 메모리에 있기 때문에, 매번 페이지 테이블을 거치면 메모리 접근이 **두 번**(테이블 1회 + 실제 데이터 1회) 필요하다. 이를 줄이기 위해 **TLB** 를 사용한다.

**TLB(Translation Look-aside Buffer)** 는 페이지 번호와 프레임 번호 쌍을 저장하는 빠른 캐시(연관 메모리)다.

```
논리 주소 발생
   │
   ▼
TLB 확인 ──(있음, TLB hit)──▶ 프레임 번호 바로 사용 ✓ 빠름
   │
   (없음, TLB miss)
   │
   ▼
페이지 테이블에서 다시 탐색 → 프레임 번호 획득
```

### 유효 접근 시간 (EAT)

TLB 적중률(hit ratio)을 $\alpha$, TLB 탐색 시간을 $\varepsilon$, 메모리 접근 시간을 1이라 하면:

$$EAT = (1 + \varepsilon)\alpha + (2 + \varepsilon)(1 - \alpha) = 2 + \varepsilon - \alpha$$

적중률 $\alpha$ 가 1에 가깝고 $\varepsilon$ 이 작을수록 EAT는 1(이상적)에 가까워진다.

> 💡 TLB 적중률이 높을수록 메모리 접근이 한 번으로 끝나 성능이 좋아진다.

> 🖼️ **이미지 추천:** 논리주소 → TLB(hit/miss 분기) → 페이지 테이블 → 물리주소로 이어지는 흐름도.

## 가상 메모리와 요구 페이징

**가상 메모리(Virtual Memory)** 는 실제 물리 메모리보다 큰 메모리를 사용하는 것처럼 보이게 하는 기술이다. 프로그램 전체를 메모리에 올리지 않고, **당장 필요한 부분만** 올린다.

**요구 페이징(Demand Paging):** 페이지가 실제로 필요해질 때(요구될 때)만 메모리로 불러오는 방식이다.

- 페이지 테이블의 각 항목에 **유효-무효 비트(valid-invalid bit)** 를 둔다
- **valid:** 해당 페이지가 메모리에 있음
- **invalid:** 해당 페이지가 메모리에 없음 (디스크에 있음)

> 💡 가상 메모리 덕분에 메모리보다 큰 프로그램도 실행할 수 있고, 더 많은 프로세스를 동시에 올릴 수 있다.

## 페이지 폴트 · 히트 · 미스

CPU가 접근하려는 페이지가 메모리에 있는지 여부에 따라 나뉜다.

| 용어 | 설명 |
| --- | --- |
| **페이지 히트(Page Hit)** | 접근하려는 페이지가 이미 메모리에 있음 → 바로 사용 |
| **페이지 미스 = 페이지 폴트(Page Fault)** | 접근하려는 페이지가 메모리에 없음 → 디스크에서 불러와야 함 |

### 페이지 폴트 처리 과정

```
1. CPU가 페이지 접근 → 페이지 테이블 확인
2. valid-invalid bit가 invalid → 페이지 폴트 발생!
3. 운영체제가 트랩으로 제어권을 가져감
4. 디스크(backing store)에서 해당 페이지를 찾음
5. 빈 프레임에 페이지를 올림 (빈 프레임이 없으면 페이지 교체)
6. 페이지 테이블 갱신 (valid로 변경)
7. 중단됐던 명령어를 다시 실행
```

> 💡 페이지 폴트는 에러가 아니다. 요구 페이징에서 자연스럽게 발생하는 정상 동작이며, 다만 디스크 접근이 필요해 시간이 오래 걸린다. 페이지 폴트가 너무 잦으면 성능이 급격히 떨어진다.

> 🖼️ **이미지 추천:** 페이지 폴트 발생 → 트랩 → 디스크에서 페이지 로드 → 페이지 테이블 갱신 → 명령어 재실행으로 이어지는 단계별 흐름도.

## 페이지 교체 알고리즘

메모리에 빈 프레임이 없을 때, **어떤 페이지를 내보내고(victim)** 새 페이지를 올릴지 결정하는 알고리즘이다. 목표는 **페이지 폴트 횟수를 최소화**하는 것이다.

| 알고리즘 | 교체 기준 | 특징 |
| --- | --- | --- |
| **OPT (LFD)** | 앞으로 **가장 오랫동안 사용되지 않을** 페이지 | 이론상 최적, 미래를 알아야 해서 실제 구현 불가 (기준점으로만 사용) |
| **FIFO** | **가장 먼저 들어온** 페이지 | 구현이 단순. 단 Belady's Anomaly 발생 가능 |
| **LRU** | **가장 오랫동안 사용되지 않은** 페이지 (과거 기준) | OPT의 현실적 근사. 성능 좋음 |
| **NUR** | 최근에 사용되지 않은 페이지 (참조 비트 활용) | LRU의 근사. 오버헤드 적음 |
| **LFU** | **사용 빈도가 가장 낮은** 페이지 | 참조 횟수를 카운트 |

> 💡 **OPT(LFD, Longest Forward Distance)** 는 "앞으로 가장 늦게 쓰일 페이지를 교체"하는 방식으로 페이지 폴트가 가장 적다. 하지만 미래의 참조를 미리 알아야 하므로 실제로는 구현할 수 없고, 다른 알고리즘의 성능을 비교하는 기준으로만 쓰인다.

> 💡 **LRU(Least Recently Used)** 는 "과거에 가장 오래 안 쓰인 페이지가 미래에도 안 쓰일 것"이라고 가정한다. OPT를 과거 방향으로 뒤집은 현실적 대안으로 가장 널리 쓰인다.

> 💡 **Belady's Anomaly:** FIFO에서는 프레임 수를 늘렸는데 오히려 페이지 폴트가 증가하는 역설적 현상이 나타날 수 있다. LRU·OPT에서는 발생하지 않는다.

> 🖼️ **이미지 추천:** 같은 페이지 참조 순서에 대해 FIFO / LRU / OPT가 각각 어떤 페이지를 교체하는지 비교하는 표 형태의 시뮬레이션 그림.

## 스레싱

**스레싱(Thrashing)** 은 페이지 폴트가 지나치게 자주 발생해, CPU가 실제 작업보다 **페이지 교체에만 시간을 다 쓰는** 상태다.

```
프로세스 수를 늘림 → 각 프로세스에 할당된 프레임 부족
   → 페이지 폴트 폭증 → 디스크 입출력에 매달림
   → CPU 이용률 급락 → (더 올리려고 프로세스 추가) → 악화
```

```
CPU
이용률 │      ┌──────╮
       │     /        ╲  ← 이 지점을 넘으면 스레싱
       │    /          ╲
       │   /            ╲___
       │  /                 ╲___
       └─────────────────────────▶ 다중 프로그래밍 정도
                              (메모리에 올린 프로세스 수)
```

- 다중 프로그래밍 정도(동시 실행 프로세스 수)를 높이면 CPU 이용률이 올라가지만, 어느 지점을 넘으면 급격히 떨어진다
- 각 프로세스가 필요한 만큼의 프레임을 확보하지 못해서 발생한다

> 💡 **해결 방법**
> 워킹 셋(Working Set) 모델이나 페이지 폴트 빈도(PFF, Page-Fault Frequency)를 이용해, 각 프로세스에 필요한 만큼의 프레임을 보장하고 다중 프로그래밍 정도를 조절한다.

> 🖼️ **이미지 추천:** 다중 프로그래밍 정도(x축)에 따른 CPU 이용률(y축) 그래프 — 어느 지점에서 급락하는 스레싱 구간을 표시.
