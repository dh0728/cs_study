# 6주차: 네트워크 - TCP/IP + 네트워크 기기 + IP 주소 + HTTP

---

## 전체 관계

```
데이터가 인터넷을 통해 전달되는 과정

[ 내 컴퓨터 ]
    ↓ HTTP로 요청 작성
    ↓ TCP/IP 4계층을 거쳐 포장
[ 네트워크 기기들 (허브/스위치/라우터 등) ]
    ↓ IP 주소로 목적지 찾아감 (홉바이홉)
[ 상대 서버 ]
    ↓ 4계층을 거쳐 풀어서 읽음
    ↓ HTTP로 응답
```

> TCP/IP = 인터넷에서 데이터를 주고받는 **공통 규칙(프로토콜)**

---

# 1. TCP/IP 4계층 모델

## 1-1. 계층 구조

### 한 줄 정의
> 데이터를 보낼 때 **4단계로 나눠서 처리**하는 구조

### 비유
> 택배 보내기
> - 물건 담기 (응용)
> - 포장하기 (전송)
> - 주소 붙이기 (인터넷)
> - 트럭에 싣기 (네트워크 접근)

### 4계층 구조

```
┌─────────────────────────────┐
│  4계층: 응용 계층            │  앱이 직접 쓰는 층
│  (Application Layer)        │  HTTP, FTP, DNS, SMTP
├─────────────────────────────┤
│  3계층: 전송 계층            │  데이터 신뢰성 담당
│  (Transport Layer)          │  TCP, UDP
├─────────────────────────────┤
│  2계층: 인터넷 계층          │  목적지 주소 담당
│  (Internet Layer)           │  IP, ARP, ICMP
├─────────────────────────────┤
│  1계층: 네트워크 접근 계층   │  물리적 전송 담당
│  (Network Access Layer)     │  이더넷, Wi-Fi
└─────────────────────────────┘
```

### 각 계층 설명

| 계층 | 역할 | 주요 프로토콜 | 비유 |
|---|---|---|---|
| 응용 계층 | 사람이 쓰는 앱과 연결 | HTTP, FTP, DNS | 편지 내용 쓰기 |
| 전송 계층 | 데이터 정확히 전달 보장 | TCP, UDP | 편지 봉투에 넣기 |
| 인터넷 계층 | 목적지 찾아서 이동 | IP, ARP | 주소 적기 |
| 네트워크 접근 계층 | 실제 물리적 전송 | 이더넷, Wi-Fi | 우체통에 넣기 |

### 데이터 흐름

```
보내는 쪽 (캡슐화):          받는 쪽 (역캡슐화):
응용 계층   → 데이터         데이터   → 응용 계층
전송 계층   → 세그먼트       세그먼트 → 전송 계층
인터넷 계층 → 패킷           패킷     → 인터넷 계층
네트워크    → 프레임         프레임   → 네트워크
    ↓                            ↑
        물리적 전송 (케이블/무선)
```

> **캡슐화** = 보낼 때 각 층에서 헤더(주소/정보)를 붙이는 것
> **역캡슐화** = 받을 때 각 층에서 헤더를 떼어내는 것

---

## 1-2. PDU (Protocol Data Unit)

### 한 줄 정의
> 각 계층에서 데이터를 부르는 **이름**

### 비유
> 같은 편지인데 단계마다 이름이 달라짐
> - 내용물 → 봉투 → 박스 → 컨테이너

### 계층별 PDU 이름

| 계층 | PDU 이름 | 설명 |
|---|---|---|
| 응용 계층 | 메시지 (Message) | 사람이 보내는 데이터 원본 |
| 전송 계층 | 세그먼트 (Segment) / 데이터그램 | TCP면 세그먼트, UDP면 데이터그램 |
| 인터넷 계층 | 패킷 (Packet) | IP 주소 붙은 상태 |
| 네트워크 접근 계층 | 프레임 (Frame) | 물리 주소(MAC) 붙은 상태 |

```
[ 메시지 ]
    ↓ 전송 계층에서 헤더 추가
[ TCP 헤더 | 메시지 ]  = 세그먼트
    ↓ 인터넷 계층에서 헤더 추가
[ IP 헤더 | TCP 헤더 | 메시지 ]  = 패킷
    ↓ 네트워크 계층에서 헤더 추가
[ MAC 헤더 | IP 헤더 | TCP 헤더 | 메시지 ]  = 프레임
```

---

## 1-3. TCP vs UDP

### TCP (Transmission Control Protocol)

### 한 줄 정의
> **신뢰성** 있는 전송. 데이터가 반드시 도착함을 보장

### 동작 방식: 3-way Handshake

```
클라이언트          서버
    │── SYN ──────→│   "연결할게"
    │←─ SYN+ACK ──│   "OK, 나도 연결할게"
    │── ACK ──────→│   "확인"
    │                  연결 완료!
    │── 데이터 ───→│
    │←─ ACK ──────│   "잘 받았어"
```

| 특징 | 설명 |
|---|---|
| 연결 지향 | 보내기 전에 연결 먼저 |
| 신뢰성 | 데이터 순서 보장, 손실 시 재전송 |
| 속도 | 느림 (확인 과정 있음) |
| 사용 예 | 웹 (HTTP), 이메일, 파일 전송 |

---

### UDP (User Datagram Protocol)

### 한 줄 정의
> **빠른** 전송. 도착 보장 안 함. 그냥 던짐

```
클라이언트          서버
    │── 데이터 ───→│   확인 없이 그냥 보냄
    │── 데이터 ───→│   도착 여부 모름
```

| 특징 | 설명 |
|---|---|
| 비연결 지향 | 연결 과정 없음 |
| 신뢰성 없음 | 손실돼도 재전송 안 함 |
| 속도 | 빠름 |
| 사용 예 | 스트리밍, 게임, 화상통화, DNS |

### TCP vs UDP 비교

| | TCP | UDP |
|---|---|---|
| 연결 | 연결 후 전송 | 그냥 전송 |
| 신뢰성 | 높음 (보장) | 낮음 (보장 없음) |
| 속도 | 느림 | 빠름 |
| 비유 | 등기우편 (수신 확인) | 일반우편 (그냥 넣음) |
| 사용 | 파일전송, 웹, 이메일 | 영상통화, 게임, 방송 |

# 2. 네트워크 기기

## 2-1. 네트워크 기기의 처리 범위

### 한 줄 정의
> 네트워크 기기는 **처리할 수 있는 계층이 각각 다름**

### 핵심 원칙
> 상위 계층 처리 가능한 기기 = 하위 계층도 처리 가능

```
계층          기기
응용 계층  →  L7 스위치 (= 애플리케이션 게이트웨이)
인터넷 계층 → 라우터, L3 스위치
데이터 링크 → L2 스위치, 브리지
물리 계층  →  NIC, 리피터, AP (허브)
```

---

## 2-2. 애플리케이션 계층을 처리하는 기기

### L7 스위치 (= 로드 밸런서)

### 한 줄 정의
> 가장 상위 계층(응용)까지 이해하는 기기. **트래픽 분산** 담당

### 비유
> 대형 식당 입구 직원. 손님을 빈 테이블로 안내

### 역할

| 기능 | 설명 |
|---|---|
| 로드 밸런싱 | 서버 여러 대에 요청을 골고루 분산 |
| 헬스 체크 | 서버가 살아있는지 주기적으로 확인 |
| URL 기반 라우팅 | /image → 이미지 서버, /api → API 서버 |

```
클라이언트 요청
      ↓
  [ L7 스위치 ]
   ↙    ↓    ↘
서버A  서버B  서버C   ← 트래픽 골고루 분산
```

---

## 2-3. 인터넷 계층을 처리하는 기기

### 라우터 (Router)

### 한 줄 정의
> **IP 주소** 보고 패킷을 다음 목적지로 전달하는 기기

### 비유
> 도로 이정표. "부산 가려면 이쪽으로"

### 역할
- 라우팅 테이블 보고 최적 경로 결정
- 서로 다른 네트워크 연결 (집 ↔ 인터넷)
- 홉바이홉 통신의 핵심 기기

### L3 스위치

| | 라우터 | L3 스위치 |
|---|---|---|
| 처리 계층 | 인터넷 계층 | 인터넷 계층 |
| 속도 | 보통 | 빠름 (하드웨어 처리) |
| 사용 위치 | 네트워크 경계 | 내부 네트워크 |

---

## 2-4. 데이터 링크 계층을 처리하는 기기

### L2 스위치

### 한 줄 정의
> **MAC 주소** 보고 같은 네트워크 안에서 데이터 전달하는 기기

### 비유
> 아파트 내부 우편함. 같은 건물 안 배달

### 역할
- MAC 주소 테이블 관리
- 목적지 MAC 주소 있으면 → 해당 포트로만 전송
- 목적지 모르면 → 전체 포트에 뿌림 (플러딩)

### 브리지 (Bridge)

| | L2 스위치 | 브리지 |
|---|---|---|
| 포트 수 | 많음 | 적음 (2~3개) |
| 처리 속도 | 빠름 (하드웨어) | 느림 (소프트웨어) |
| 사용 | 현재 주로 사용 | 거의 안 씀 |

> 둘 다 MAC 주소 기반으로 같은 네트워크 내 전달 담당

---

## 2-5. 물리 계층을 처리하는 기기

### NIC (Network Interface Card, 랜카드)

### 한 줄 정의
> 컴퓨터에 내장된 **네트워크 연결 장치**. MAC 주소 가짐

- 비유: 컴퓨터의 귀와 입
- 모든 네트워크 통신의 시작점

### 리피터 (Repeater)

### 한 줄 정의
> 신호가 약해지면 **증폭해서 다시 보내는** 기기

- 비유: 마라톤 중계 지점에서 선수에게 물 주는 것
- 현재: 거의 안 씀 (스위치가 대체)

### AP (Access Point, 공유기)

### 한 줄 정의
> **유선 ↔ 무선 변환** + 여러 기기를 네트워크에 연결

- 비유: 무선 신호 뿌리는 기지국
- 집에서 쓰는 공유기 = AP + 라우터 기능 합쳐진 것

### 물리 계층 기기 비교

| 기기 | 역할 | 현재 사용 |
|---|---|---|
| NIC | 네트워크 연결 | 모든 기기에 내장 |
| 리피터 | 신호 증폭 | 거의 안 씀 |
| AP | 무선 연결 | 현재 주력 |

---

# 3. IP 주소

## 3-1. IP 주소란?

### 한 줄 정의
> 인터넷에서 컴퓨터를 찾기 위한 **주소**

### 비유
> 집 주소. 없으면 택배 못 보냄

---

## 3-2. IP 주소 체계

### IPv4

```
192 . 168 . 0 . 1
 ↑     ↑    ↑   ↑
 8비트 8비트 8비트 8비트  = 총 32비트

각 자리: 0~255
```

| 항목 | 설명 |
|---|---|
| 형식 | 숫자 4개를 점(.)으로 구분 |
| 범위 | 0.0.0.0 ~ 255.255.255.255 |
| 총 개수 | 약 43억 개 |
| 문제 | 43억 개로 부족 → IPv6 등장 |

### IPv6

```
2001:0db8:85a3:0000:0000:8a2e:0370:7334
```

| 항목 | 설명 |
|---|---|
| 형식 | 16진수 8그룹, 콜론(:)으로 구분 |
| 총 개수 | 약 340조 × 1조 × 1조 개 (사실상 무한) |
| 현재 | IPv4와 병행 사용 중 |

### 공인 IP vs 사설 IP

| | 공인 IP | 사설 IP |
|---|---|---|
| 뜻 | 인터넷에서 쓰는 진짜 주소 | 집/회사 내부에서 쓰는 주소 |
| 범위 | 전 세계 유일 | 192.168.x.x 등 |
| 비유 | 집의 도로명 주소 | 집 안 방 번호 |
| 할당 | ISP(통신사)가 줌 | 공유기가 줌 |

### 서브넷 마스크

```
IP 주소:      192.168.  0.  1
서브넷 마스크: 255.255.255.  0

→ 앞 24비트 = 네트워크 주소 (192.168.0)
→ 뒤  8비트 = 호스트 주소 (.1)
```

> 서브넷 마스크 = IP 주소에서 **어디까지가 네트워크, 어디부터가 장치 번호**인지 구분

---

## 3-3. ARP (Address Resolution Protocol)

### 한 줄 정의
> **IP 주소 → MAC 주소** 로 바꿔주는 프로토콜

### 왜 필요한가?
- 인터넷 계층은 IP 주소 사용
- 네트워크 접근 계층은 MAC 주소 사용
- 실제 전송하려면 MAC 주소 필요 → ARP로 변환

### MAC 주소란?
> 네트워크 장치(랜카드 등)에 **공장에서 박아 넣은 고유 번호**
> - 비유: 주민등록번호 (변경 불가)
> - 형식: `AA:BB:CC:DD:EE:FF` (16진수 6자리)

### ARP 동작 과정

```
나: "192.168.0.5 MAC 주소가 뭐야?"  → 네트워크 전체에 물어봄 (브로드캐스트)

192.168.0.5: "나야! 내 MAC은 AA:BB:CC:DD:EE:FF"  → 응답

나: MAC 주소 저장 (ARP 캐시) → 이제 직접 전송 가능
```

| 용어 | 설명 |
|---|---|
| 브로드캐스트 | 같은 네트워크 내 전체에 뿌리기 |
| ARP 캐시 | 알아낸 MAC 주소 임시 저장 |

---

## 3-4. 홉바이홉 통신 (Hop by Hop)

### 한 줄 정의
> 데이터가 목적지까지 **라우터를 하나씩 거쳐** 이동하는 것

### 비유
> 서울 → 부산 가는 버스
> - 서울 → 대전 (1홉)
> - 대전 → 대구 (2홉)
> - 대구 → 부산 (3홉)
> 직통 없이 중간 정류장 거침

### 동작 과정

```
내 PC → 공유기 → ISP 라우터 → ... → 목적지 서버
  (1홉)   (2홉)      (3홉)    (n홉)
```

### 핵심 용어

| 용어 | 설명 |
|---|---|
| 홉 (Hop) | 라우터 하나를 거치는 것 |
| 라우터 (Router) | 다음 목적지로 패킷을 전달하는 장치 |
| 라우팅 테이블 | 라우터가 가진 "어디로 보낼지" 지도 |
| TTL (Time To Live) | 패킷이 살 수 있는 홉 횟수. 0되면 소멸 |

### 라우팅 테이블

```
라우터 A의 라우팅 테이블:
목적지 IP       다음 라우터
10.0.0.0/8  →  라우터 B
192.168.0.0 →  라우터 C
0.0.0.0     →  기본 게이트웨이
```

> 라우터는 패킷을 받을 때마다 테이블 보고 "다음은 어디로?" 결정

---

## 3-5. IP 주소를 이용한 위치 정보

### 한 줄 정의
> IP 주소로 **대략적인 위치(국가/도시)** 를 추정하는 것

### 어떻게 되나?

```
IP 주소 → IP 데이터베이스 조회 → 국가/도시/ISP 정보 반환
```

### 비유
> 우편번호로 어느 지역인지 아는 것

### 활용 예시

| 서비스 | 활용 방식 |
|---|---|
| 넷플릭스 | IP로 국가 판단 → 해당 국가 콘텐츠만 제공 |
| 구글 | IP로 언어/지역 자동 설정 |
| 게임 서버 | IP로 가장 가까운 서버 연결 |
| 보안 | 해외 IP 접속 시 추가 인증 요구 |

### 한계
- 정확한 주소 (동/호수) 는 모름
- VPN 쓰면 다른 나라 IP처럼 위장 가능
- ISP 따라 정확도 다름

# 4. HTTP

## 4-1. HTTP란?

### 한 줄 정의
> 웹에서 데이터를 주고받는 **규칙(프로토콜)**

### 비유
> 손님(브라우저)과 식당(서버) 사이의 **주문 언어**

```
브라우저: "GET /index.html"  → 요청 (Request)
서버:     "200 OK + 데이터"  → 응답 (Response)
```

---

## 4-2. HTTP/1.0

### 한 줄 정의
> 요청 하나 → 연결 하나. **매번 새로 연결**

### 문제점

```
요청1: 연결 → 요청 → 응답 → 연결 끊음
요청2: 연결 → 요청 → 응답 → 연결 끊음   ← 매번 반복
요청3: 연결 → 요청 → 응답 → 연결 끊음
```

- 매 요청마다 3-way Handshake 반복 → **느림**
- RTT (Round Trip Time) 증가

> **RTT** = 요청 보내고 응답 받는 데 걸리는 시간

---

## 4-3. HTTP/1.1

### 한 줄 정의
> **한 번 연결하면 여러 요청** 가능 (keep-alive)

### 개선점

```
연결 한 번
    ├→ 요청1 → 응답1
    ├→ 요청2 → 응답2   ← 연결 유지
    └→ 요청3 → 응답3
연결 끊음
```

### 핵심 개선

| 기능 | 설명 |
|---|---|
| keep-alive | 연결 유지. 매번 새로 안 해도 됨 |
| 파이프라이닝 | 응답 기다리지 않고 요청 여러 개 연속 전송 |
| 호스트 헤더 | 하나의 IP에 여러 도메인 운영 가능 |

### 남은 문제점: HOL Blocking

```
요청1 (큰 파일) ──────────────────→ 응답1
요청2 (작은 파일) ─────────────────────→ 응답2  ← 요청1 끝날 때까지 기다림
```

> **HOL Blocking** (Head Of Line Blocking) = 앞 요청이 막히면 뒤 요청도 줄줄이 막히는 현상

---

## 4-4. HTTP/2

### 한 줄 정의
> 여러 요청을 **동시에** 처리. HOL Blocking 해결

### 핵심 개선

#### 멀티플렉싱 (Multiplexing)

```
HTTP/1.1:          HTTP/2:
요청1 ──→ 응답1     요청1 ──→ 응답1
요청2 ──→ 응답2     요청2 ──→ 응답2   ← 동시 처리
요청3 ──→ 응답3     요청3 ──→ 응답3
(순서대로)          (순서 상관없이)
```

| 기능 | 설명 |
|---|---|
| 멀티플렉싱 | 하나의 연결로 여러 요청 동시 처리 |
| 헤더 압축 | 중복 헤더 제거 → 데이터 크기 줄임 |
| 서버 푸시 | 요청 안 해도 서버가 미리 보내줌 |
| 바이너리 전송 | 텍스트 → 바이너리로 변환. 처리 빠름 |

---

## 4-5. HTTPS

### 한 줄 정의
> HTTP + **암호화 (TLS/SSL)**

### 비유
> 편지를 암호로 써서 보냄. 중간에 누가 훔쳐봐도 못 읽음

### HTTP vs HTTPS

| | HTTP | HTTPS |
|---|---|---|
| 암호화 | 없음 | 있음 (TLS) |
| 포트 | 80 | 443 |
| 주소 | http:// | https:// |
| 자물쇠 | 없음 | 브라우저에 표시됨 |

### TLS Handshake (간략)

```
클라이언트                서버
    │── 지원하는 암호화 목록 →│
    │←─ 인증서 + 선택한 방식 ─│
    │── 인증서 검증 + 키 교환 →│
    │←─────── 암호화 통신 ────│
```

### 왜 HTTPS 써야 하는가?
- 개인정보 (비밀번호, 카드번호) 보호
- 중간자 공격 (내용 가로채기) 방어
- 구글 검색 순위에도 영향

---

## 4-6. HTTP/3

### 한 줄 정의
> TCP 대신 **UDP 기반 (QUIC)** 으로 더 빠르게

### 왜 UDP 쓰나?
- HTTP/2까지는 TCP 사용
- TCP 자체의 HOL Blocking이 남아있음
- UDP로 바꾸고 신뢰성은 QUIC이 직접 처리

### QUIC이란?
> Google이 만든 새 프로토콜. UDP 위에서 TCP처럼 신뢰성 제공

```
HTTP/1.1, 2:  [HTTP] → [TCP] → [IP]
HTTP/3:       [HTTP] → [QUIC] → [UDP] → [IP]
```

### 핵심 개선

| 기능 | 설명 |
|---|---|
| 연결 속도 | 1-RTT 또는 0-RTT. TCP보다 빠른 연결 |
| HOL Blocking 완전 해결 | 스트림 독립. 하나 막혀도 다른 것 진행 |
| 연결 유지 | IP 바뀌어도 연결 유지 (모바일 이동 시) |

### HTTP 버전 비교

| 버전 | 전송 방식 | 핵심 개선 |
|---|---|---|
| HTTP/1.0 | TCP, 매번 연결 | 최초 |
| HTTP/1.1 | TCP, 연결 유지 | keep-alive, 파이프라이닝 |
| HTTP/2 | TCP, 멀티플렉싱 | 동시 처리, 헤더 압축 |
| HTTPS | TCP + TLS | 암호화 |
| HTTP/3 | UDP (QUIC) | 더 빠른 연결, HOL 완전 해결 |

## 전체 요약

| 항목 | 핵심 한 줄 |
|---|---|
| TCP/IP 4계층 | 데이터를 4단계로 나눠 포장하고 전송하는 구조 |
| PDU | 계층마다 데이터 이름 다름. 메시지→세그먼트→패킷→프레임 |
| TCP | 연결 후 전송. 신뢰성 높음. 느림 |
| UDP | 그냥 전송. 신뢰성 없음. 빠름 |
| L7 스위치 | 응용 계층 처리. 트래픽 분산 (로드 밸런서) |
| 라우터 | 인터넷 계층. IP 주소 보고 경로 결정 |
| L2 스위치 | 데이터 링크. MAC 주소 보고 같은 네트워크 내 전달 |
| NIC / AP | 물리 계층. 네트워크 연결 / 무선 신호 |
| IPv4 | 32비트 주소. 약 43억 개 |
| IPv6 | 128비트 주소. 사실상 무한 |
| ARP | IP 주소 → MAC 주소 변환 |
| 홉바이홉 | 라우터 하나씩 거쳐 목적지까지 이동 |
| IP 위치 정보 | IP로 국가/도시 추정. 정확한 주소는 모름 |
| HTTP/1.0 | 매번 새 연결. 느림 |
| HTTP/1.1 | 연결 유지. HOL Blocking 문제 남음 |
| HTTP/2 | 멀티플렉싱. 동시 처리 |
| HTTPS | HTTP + TLS 암호화 |
| HTTP/3 | UDP(QUIC) 기반. 가장 빠름 |
