# 5주차: 프로세스와 스레드

---

## 전체 관계

```
프로그램 (파일, 저장된 상태)
    ↓ 실행하면
프로세스 (실행 중인 상태)
    ↓ 안에 있는 것
스레드 (실제로 일하는 단위)
```

> 프로세스 = 공장 / 스레드 = 공장 안 일꾼

---

# 1. 프로세스 (Process)

## 1-1. 프로세스란?

### 한 줄 정의
> **실행 중인 프로그램**

### 비유

| | 비유 |
|---|---|
| 프로그램 | 요리 레시피 (종이) |
| 프로세스 | 레시피대로 실제 요리 중인 상태 |

- 크롬 더블클릭 → 프로세스 생성
- 크롬 여러 개 열기 → 프로세스 여러 개

### 프로세스가 가진 것

```
┌─────────────────────┐
│      프로세스        │
├─────────────────────┤
│  코드 (Code)         │  실행할 명령어들
│  데이터 (Data)       │  전역 변수
│  힙 (Heap)           │  동적으로 할당한 메모리
│  스택 (Stack)        │  함수 호출, 지역 변수
└─────────────────────┘
```

---

## 1-2. 프로세스 상태

### 한 줄 정의
> 프로세스는 살아있는 동안 5가지 상태를 오감

### 상태 흐름

```
생성 → 준비 → 실행 → 대기 → 종료
               ↑  ↓
              준비 ← 대기
```

| 상태 | 설명 | 비유 |
|---|---|---|
| 생성 (New) | 프로세스 막 만들어짐 | 손님이 식당에 들어옴 |
| 준비 (Ready) | CPU 쓸 차례 기다리는 중 | 자리 배정 기다리는 중 |
| 실행 (Running) | CPU가 지금 처리 중 | 요리 나와서 먹는 중 |
| 대기 (Waiting) | 뭔가 기다리는 중 (파일 읽기 등) | 추가 주문 기다리는 중 |
| 종료 (Terminated) | 다 끝남 | 식사 마치고 나감 |

---

## 1-3. PCB (프로세스 제어 블록)

### 한 줄 정의
> OS가 프로세스 하나마다 관리하는 **정보 카드**

### 비유
> 병원 환자 차트. 환자(프로세스)마다 상태, 기록 저장

### PCB에 담긴 것

| 항목 | 설명 |
|---|---|
| 프로세스 ID (PID) | 프로세스 고유 번호 |
| 프로세스 상태 | 지금 준비/실행/대기 중 |
| 프로그램 카운터 | 다음에 실행할 명령어 위치 |
| CPU 레지스터 값 | 잠깐 멈출 때 저장해두는 값 |
| 메모리 정보 | 어느 메모리 쓰고 있는지 |
| 우선순위 | 스케줄링에 쓰임 |

---

## 1-4. 컨텍스트 스위칭 (Context Switching)

### 한 줄 정의
> CPU가 프로세스 A 하다가 B로 바꿀 때 → **A 상태 저장 + B 상태 불러오기**

### 비유
> 책 읽다가 전화 받을 때:
> 1. 책 몇 페이지 읽었는지 메모 (A 상태 저장 → PCB에)
> 2. 전화 받기 (B 실행)
> 3. 전화 끊고 메모 보고 이어 읽기 (A 상태 복원)

### 흐름

```
프로세스 A 실행 중
    ↓ 인터럽트 발생
A 상태 → PCB_A에 저장
    ↓
PCB_B에서 B 상태 불러옴
    ↓
프로세스 B 실행
    ↓ 또 인터럽트
B 상태 → PCB_B에 저장
A 상태 복원 → A 이어 실행
```

### 주의
- 컨텍스트 스위칭은 **비용(시간)이 듦**
- 너무 자주 바꾸면 → 실제 일하는 시간보다 바꾸는 시간이 더 많아짐

---

# 2. 스레드 (Thread)

## 2-1. 스레드란?

### 한 줄 정의
> 프로세스 안에서 **실제로 일하는 단위**

### 비유

| | 비유 |
|---|---|
| 프로세스 | 공장 |
| 스레드 | 공장 안 일꾼 |

- 일꾼 1명 = 싱글 스레드
- 일꾼 여러 명 = 멀티 스레드

### 스레드가 가진 것

```
┌────────────────────────────────┐
│           프로세스              │
│  코드 / 데이터 / 힙  (공유)    │
│  ┌──────────┐ ┌──────────┐    │
│  │ 스레드 1  │ │ 스레드 2  │   │
│  │ 스택     │ │ 스택     │    │
│  │ 레지스터 │ │ 레지스터 │    │
│  └──────────┘ └──────────┘    │
└────────────────────────────────┘
```

- 스레드끼리 **코드/데이터/힙은 공유**
- **스택/레지스터는 각자 따로**

---

## 2-2. 프로세스 vs 스레드

| | 프로세스 | 스레드 |
|---|---|---|
| 단위 | 실행 중인 프로그램 | 프로세스 안 작업 단위 |
| 메모리 | 각자 독립적 | 프로세스 안에서 공유 |
| 생성 비용 | 무거움 (느림) | 가벼움 (빠름) |
| 충돌 시 | 다른 프로세스 안 죽음 | 한 스레드 죽으면 전체 위험 |
| 통신 | 어려움 (IPC 필요) | 쉬움 (메모리 공유) |
| 비유 | 각자 다른 공장 | 같은 공장 안 일꾼들 |

---

## 2-3. 멀티프로세스 vs 멀티스레드

### 멀티프로세스
> 같은 작업을 여러 **프로세스**로 나눠서 처리

- 예) 크롬 탭마다 프로세스 하나
- 장점: 한 탭 죽어도 다른 탭 살아있음
- 단점: 메모리 많이 씀

### 멀티스레드
> 하나의 프로세스 안에서 여러 **스레드**로 나눠서 처리

- 예) 유튜브 = 영상 재생 + 댓글 로딩 + 광고 → 동시에
- 장점: 메모리 적게 씀, 빠름
- 단점: 스레드 하나 오류나면 전체 위험

### 비유 비교

| | 멀티프로세스 | 멀티스레드 |
|---|---|---|
| 비유 | 공장 여러 개 | 공장 하나에 일꾼 여러 명 |
| 안전성 | 높음 | 낮음 |
| 속도 | 느림 | 빠름 |
| 메모리 | 많이 씀 | 적게 씀 |

# 3. CPU 스케줄링

## 3-1. CPU 스케줄링이란?

### 한 줄 정의
> **여러 프로세스 중 누구한테 CPU를 줄지 결정하는 것**

### 왜 필요한가?
- CPU는 한 번에 하나만 처리 가능
- 프로세스는 여러 개 동시에 실행 원함
- OS가 순서를 정해줘야 함

### 비유
> 은행 창구 1개 / 손님 여러 명 → **누구 먼저 처리할지 정하는 규칙**

---

## 3-2. 스케줄링 알고리즘

### 1) FCFS (First Come First Served)
> 먼저 온 프로세스 먼저 처리

```
도착 순서: A → B → C
처리 순서: A → B → C
```

| 장점 | 단점 |
|---|---|
| 구현 쉬움 | 긴 작업이 앞에 오면 뒤가 오래 기다림 |

- **비유**: 편의점 계산대 줄 서기

---

### 2) SJF (Shortest Job First)
> 처리 시간 **제일 짧은** 프로세스 먼저

```
작업 시간: A=10초, B=2초, C=5초
처리 순서: B(2) → C(5) → A(10)
```

| 장점 | 단점 |
|---|---|
| 평균 대기시간 최소 | 긴 작업은 영원히 못 할 수도 있음 (기아 현상) |

- **비유**: 마트 계산대에서 물건 적은 사람 먼저

---

### 3) Round Robin (RR)
> 모두에게 **동일한 시간(타임 퀀텀)** 만큼 돌아가면서 줌

```
타임 퀀텀 = 3초
A(10초), B(4초), C(6초)

A 3초 → B 3초 → C 3초 → A 3초 → B 1초 → C 3초 → A 3초 → A 1초
```

| 장점 | 단점 |
|---|---|
| 공평함. 아무도 굶지 않음 | 타임 퀀텀 설정이 중요 |

- **비유**: 놀이기구 줄 서기. 한 번 타면 내려오고 다시 줄 서기
- 타임 퀀텀 너무 짧음 → 컨텍스트 스위칭 너무 많아짐
- 타임 퀀텀 너무 김 → FCFS랑 똑같아짐

---

### 4) Priority Scheduling (우선순위 스케줄링)
> **우선순위 높은** 프로세스 먼저

```
우선순위: A=3, B=1, C=2  (숫자 작을수록 높음)
처리 순서: B(1) → C(2) → A(3)
```

| 장점 | 단점 |
|---|---|
| 중요한 작업 빨리 처리 | 낮은 우선순위는 영원히 못 할 수도 (기아 현상) |

- **비유**: 응급실 환자 우선순위 (= 우선순위 큐!)
- 기아 현상 해결 → **에이징(Aging)**: 오래 기다릴수록 우선순위 올려줌

---

### 5) MLQ / MLFQ (다단계 큐)
> 우선순위별로 **큐(줄)를 여러 개** 만들어서 관리

```
큐 1 (높은 우선순위): 시스템 프로세스
큐 2 (중간):          일반 앱
큐 3 (낮은 우선순위): 백그라운드 작업
```

- MLQ = 큐 간 이동 불가
- MLFQ = 큐 간 이동 가능 (오래 걸리면 낮은 큐로 내려감)
- **비유**: 항공기 탑승 (비즈니스 → 일반석 순서)

---

## 3-3. 스케줄링 알고리즘 비교

| 알고리즘 | 순서 기준 | 기아 현상 | 공평성 |
|---|---|---|---|
| FCFS | 도착 순서 | 없음 | 보통 |
| SJF | 짧은 작업 먼저 | 있음 | 낮음 |
| Round Robin | 시간 순환 | 없음 | 높음 |
| 우선순위 | 우선순위 높은 것 | 있음 | 낮음 |
| MLFQ | 다단계 큐 | 에이징으로 해결 | 높음 |

> **기아 현상 (Starvation)** = 우선순위 낮은 프로세스가 CPU를 영원히 못 받는 것

---

## 전체 요약

| 항목 | 핵심 한 줄 |
|---|---|
| 프로세스 | 실행 중인 프로그램. 독립된 메모리 공간 |
| 프로세스 상태 | 생성→준비→실행→대기→종료 를 오감 |
| PCB | OS가 프로세스마다 관리하는 정보 카드 |
| 컨텍스트 스위칭 | CPU가 프로세스 바꿀 때 상태 저장/복원 |
| 스레드 | 프로세스 안 실제 일하는 단위. 메모리 공유 |
| 멀티프로세스 | 공장 여러 개. 안전하지만 무거움 |
| 멀티스레드 | 공장 하나에 일꾼 여러 명. 빠르지만 위험 |
| CPU 스케줄링 | 누구한테 CPU 줄지 결정하는 규칙 |
| FCFS | 먼저 온 순서 |
| SJF | 짧은 작업 먼저 |
| Round Robin | 시간 나눠서 돌아가며 |
| 우선순위 | 중요한 것 먼저 |
| MLFQ | 다단계 큐로 복합 관리 |
