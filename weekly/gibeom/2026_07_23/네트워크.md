# 네트워크 핵심 정리

---

## 1. 네트워크의 기초

네트워크는 노드(Node)와 링크(Link)가 서로 연결되어 있거나 연결되어 있지 않은 집합체를 의미합니다.
* **노드:** 서버, 라우터, 스위치 등 네트워크 장치
* **링크:** 유선 또는 무선 통신 회선

### 1.1 처리량과 지연 시간
좋은 네트워크는 **많은 처리량**, **짧은 지연 시간**, **적은 장애 빈도**, **우수한 보안**을 갖춘 네트워크입니다.

* **처리량 (Throughput)**
  * 링크를 통해 전달되는 단위 시간당 데이터 양 (단위: `bps`, bits per second).
  * 트래픽, 네트워크 장치 간 대역폭, 에러, 하드웨어 스펙의 영향을 받음.
* **지연 시간 (Latency)**
  * 요청이 처리되는 시간 (메시지가 두 장치 사이를 왕복하는 데 걸리는 시간).
  * 매체 타입(유선/무선), 패킷 크기, 라우터의 패킷 처리 시간에 영향을 받음.

---

### 1.2 네트워크 토폴로지와 병목 현상
네트워크 토폴로지(Topology)는 노드와 링크가 어떻게 배치되어 있는지에 대한 연결 형태입니다.

1. **트리 토폴로지 (Tree)**
   * 계층형 토폴로지. 노드의 추가/삭제가 쉬움.
   * 특정 노드에 트래픽이 집중될 경우 하위 노드에 영향을 미칠 수 있음.
2. **버스 토폴로지 (Bus)**
   * 중앙 통신 회선 하나에 여러 노드가 연결된 구조 (LAN에서 주로 사용).
   * 설치 비용이 적고 신뢰성이 우수하나, 스푸핑(Spoofing) 공격에 취약함.
3. **스타 토폴로지 (Star)**
   * 중앙 노드에 모든 노드가 연결된 구조.
   * 에러 탐지가 쉽고 패킷 충돌 가능성이 적으나, 중앙 노드 장애 시 전체 네트워크가 마비됨.
4. **링형 토폴로지 (Ring)**
   * 노드가 양옆의 노드와 연결되어 고리 형태로 통신.
   * 노드 수가 늘어도 네트워크 손실이 적으나, 회선 하나에 장애가 발생하면 전체 네트워크에 큰 영향을 줌.
5. **메시 토폴로지 (Mesh)**
   * 그물망처럼 모든 노드가 서로 연결된 구조.
   * 장애 발생 시 대체 경로가 존재하여 신뢰성이 높고 트래픽 분산이 가능하지만, 구축 및 운용 비용이 고가임.
6. **병목 현상 (Bottleneck)**
   * 네트워크 내에서 전송 속도가 가장 느린 지점으로 인해 전체 성능이 저하되는 현상.
   * 토폴로지를 분석하여 병목 지점을 찾고 회선을 증설하거나 장비를 교체해 해결.

---

### 1.3 네트워크 분류
규모(범위)에 따른 분류:
* **LAN (Local Area Network):** 근거리 통신망 (집, 사무실 등 좁은 공간). 속도가 빠르고 혼잡도가 낮음.
* **MAN (Metropolitan Area Network):** 대도시 지역 네트워크 (도시 단위). 속도 및 혼잡도는 중간 수준.
* **WAN (Wide Area Network):** 광역 네트워크 (국가, 대륙 단위). 속도가 느리고 혼잡도가 높음.

---

### 1.4 네트워크 성능 분석 명령어
네트워크 병목 현상이나 장애 발생 시 상태를 점검하기 위해 사용하는 주요 명령어입니다.

* `ping`: 대상 노드로 ICMP 패킷을 전송하여 수신 상태 및 도달 시간을 확인.
* `netstat`: 현재 접속되어 있는 네트워크 연결 상태, 라우팅 테이블, 포트 열림 여부 확인.
* `nslookup`: DNS 서버에 관련 내용을 조회하여 특정 도메인에 매핑된 IP 주소 확인.
* `tracert` (Linux: `traceroute`): 목적지 노드까지 거치는 라우터 경로(Hop)와 각 구간별 지연 시간 확인.

## 2 TCP/IP 4계층 모델

인터넷 프로토콜 스위트는 네 개의 추상화 계층으로 구성됩니다.

| 계층 (Layer) | 주요 역할 및 프로토콜 |
| :--- | :--- |
| **애플리케이션 계층** | 응용 프로그램 간 데이터 송수신 (FTP, HTTP, SSH, SMTP, DNS) |
| **전송 계층** | 송신자와 수신자 간의 신뢰성 있는 데이터 전송 중계 (TCP, UDP) |
| **인터넷 계층** | 패킷을 목적지 IP 주소까지 라우팅하여 전송 (IP, ARP, ICMP) |
| **링크 계층** | 실질적으로 데이터를 유/무선 신호로 변환하여 전달 (Ethernet, Wi-Fi) |

### 애플리케이션 및 전송 계층 핵심 프로토콜
* **TCP vs UDP**
  * **TCP:** 연결 지향형, 패킷 순서 보장, 가상회선 방식, 3-Way Handshake로 신뢰성 확보.
  * **UDP:** 비연결 지향형, 순서 보장 안 함, 수신 여부 미확인, 전송 속도가 빠름.

---

## 3 네트워크 기기

네트워크 기기는 상위 계층을 처리하는 기기가 하위 계층도 처리할 수 있으나, 반대는 불가능합니다.

* **애플리케이션 계층 기기:** `L7 스위치` (로드밸런서 - IP/포트 외에 URL, HTTP 헤더, 쿠키 등을 기반으로 트래픽 분산 및 헬스체크 수행).
* **인터넷 계층 기기:** `라우터`, `L3 스위치` (서로 다른 네트워크를 연결 및 라우팅).
* **데이터 링크 계층 기기:** `L2 스위치` (MAC 주소 테이블 관리), `브리지`.
* **물리 계층 기기:** `NIC` (네트워크 카드), `리피터` (신호 증폭), `AP`.

## 4 IP 주소 체계

### 4.1 ARP (Address Resolution Protocol)
컴퓨터 간 실제 통신은 물리적 주소인 **MAC 주소**를 기반으로 이루어집니다. **ARP**는 논리적 주소인 **IP 주소를 기반으로 상대방의 MAC 주소를 알아내는 프로토콜**입니다.

### 4.2 홉바이홉 통신 (Hop-by-Hop)
패킷이 라우팅 테이블 정보를 바탕으로 여러 라우터(Hop)를 건너가며 목적지 IP까지 전달되는 과정입니다.
* **게이트웨이 (Gateway):** 서로 다른 프로토콜이나 네트워크를 연결해 주는 관문 역할을 하는 컴퓨터/장비.

### 4.3 IP 주소 할당 및 변환
* **IPv4 vs IPv6:** IPv4는 32비트 주소 체계이며, 주소 부족 문제를 해결하기 위해 128비트 체계인 IPv6가 도입됨.
* **DHCP:** IP 주소를 수동으로 설정하지 않고 네트워크 접속 시 자동으로 IP를 할당하는 프로토콜.
* **NAT (Network Address Translation):** 사설 IP 주소를 공인 IP 주소로 변환하는 기술. 하나의 공인 IP를 이용해 여러 기기가 인터넷에 접속할 수 있도록 지원.

## 5 HTTP 프로토콜의 발전

### 5.1 HTTP/1.0
* 한 연결당 하나의 요청/응답만 처리함.
* 매 요청마다 TCP 연결을 새로 맺어 **RTT(Round Trip Time)가 증가**하는 단점 발생.
* **최적화 기법:** 이미지 스플리팅(Sprite), 코드 압축, 이미지 Base64 인코딩 등.

### 5.2 HTTP/1.1
* `Keep-Alive` 옵션을 기본 적용하여 하나의 TCP 연결로 여러 개의 객체를 송수신 가능.
* **HOL Blocking (Head-of-Line Blocking):** 앞선 요청의 처리가 지연되면 뒤따르는 요청들이 연속적으로 대기하게 되는 문제 존재.
* 헤더에 쿠키 등 많은 메타데이터가 포함되고 압축되지 않아 무게가 큼.

### 5.3 HTTP/2
* Binary Framing 레이어를 도입하여 **멀티플렉싱(Multiplexing)** 지원 (하나의 연결에서 여러 스트림을 병렬 처리).
* **허프만 코딩** 기반의 **HPACK 헤더 압축**, **서버 푸시(Server Push)** 기능 제공.

### 5.4 HTTPS
* 애플리케이션 계층과 전송 계층 사이에 **SSL/TLS 신뢰 계층**을 두어 통신 내용을 암호화.
* **CA(인증 기관)**에서 발급한 인증서를 기반으로 서버의 신뢰성을 보장.
* **대칭키 / 비대칭키 암호화 방식**을 혼합하여 키 교환(ECDHE/DHE) 및 데이터 암호화 수행.

### 5.5 HTTP/3
* TCP가 아닌 **UDP 기반의 QUIC 프로토콜** 사용.
* 3-Way Handshake 과정을 단축 및 제거하여 **초기 연결 지연 시간 감소**.
* **순방향 오류 수정(FEC)** 메커니즘을 적용하여 패킷 손실률이 높은 열악한 네트워크 환경에서도 우수한 성능 유지.
