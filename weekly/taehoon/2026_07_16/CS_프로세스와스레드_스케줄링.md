# 3.3 프로세스와 스레드 (Process & Thread)

> **한 줄 요약**: 프로세스는 "프로그램이 실행 중인 상태", 스레드는 "프로세스 안에서 일하는 작업자"다.

---

## 3.3.1 프로세스와 컴파일 과정

### 프로그램 vs 프로세스
| 구분 | 프로그램 | 프로세스 |
|------|---------|---------|
| 정의 | 하드디스크에 저장된 **정적인 코드** | 메모리에 올라가 **실행 중인 동적 상태** |
| 비유 | 레시피(책) | 레시피대로 요리하는 **요리사** |

### 컴파일 과정 (4단계)
```
소스코드(.c) → 전처리기 → 컴파일러 → 어셈블러 → 링커 → 실행파일(.exe)
     ↓           ↓           ↓          ↓         ↓
   #include   어셈블리어    기계어     라이브러리  실행 가능한
   매크로 처리  변환        (.o)       연결       파일 생성
```

| 단계 | 역할 | 설명 |
|------|------|------|
| **전처리** | #include, #define 처리 | 레시피의 "참고 자료"를 본문에 넣는 것 |
| **컴파일** | 고급언어 → 어셈블리어 | 사람 언어 → 컴퓨터가 읽을 수 있는 언어로 번역 |
| **어셈블** | 어셈블리어 → 기계어(0/1) | 번역된 내용을 진짜 컴퓨터 명령어로 변환 |
| **링크** | 여러 파일 + 라이브러리 연결 | 요리에 필요한 "반찬(라이브러리)"도 함께 포장 |

---

## 3.3.2 프로세스의 상태

프로세스는 생애주기 동안 다음 5가지 상태를 오간다.

```
[생성] → [준비] ←→ [실행] → [종료]
            ↑         ↓
            └──── [대기] ←─┘
```

| 상태 | 설명 | 비유 |
|------|------|------|
| **생성 (New)** | 프로세스가 막 만들어진 상태 | 요리사가 주방에 들어옴 |
| **준비 (Ready)** | CPU 할당만 기다리는 상태 | 주문 대기 중 (요리대기) |
| **실행 (Running)** | CPU를 할당받아 명령어 실행 중 | 실제로 요리 중 |
| **대기 (Waiting/Blocked)** | I/O(입출력) 등 완료를 기다림 | 재료 배달 기다리는 중 |
| **종료 (Terminated)** | 실행 완료, 메모리 해제 | 요리 완료, 퇴근 |

> **스케줄러의 역할**: 준비 → 실행 (CPU 할당), 실행 → 준비 (시간 만료), 실행 → 대기 (I/O 요청)

---

## 3.3.3 프로세스의 메모리 구조

프로세스는 메모리의 4개 영역을 사용한다.

```
┌─────────────────────┐  높은 주소
│        스택         │  ← 지역 변수, 함수 호출 기록 (LIFO)
│    (Stack)          │     ↓ 위로 자람
├─────────────────────┤
│                     │
│        힙           │  ← 동적 할당 (malloc, new)
│    (Heap)           │     ↑ 아래로 자람
│                     │
├─────────────────────┤
│        데이터       │  ← 전역 변수, 정적 변수 (초기화/미초기화)
│    (Data)           │
├─────────────────────┤
│        코드         │  ← 실행할 프로그램 코드 (읽기 전용)
│    (Code/Text)      │
└─────────────────────┘  낮은 주소
```

| 영역 | 특징 | 예시 |
|------|------|------|
| **코드** | 읽기 전용, 실행 코드 저장 | `main()`, `printf()` 함수 코드 |
| **데이터** | 전역/정적 변수 저장 | `int g_count = 0;` |
| **힙** | 런타임에 동적 할당, 개발자 관리 | `malloc(100)`, `new Object()` |
| **스택** | 함수 호출 시 자동 할당, 지역 변수 | `int a = 10;` (함수 안) |

> ⚠️ **스택 오버플로우**: 스택이 힙 영역을 침범할 때 (재귀함수 무한 호출 등)
> ⚠️ **힙 오버플로우**: 힙이 스택 영역을 침범할 때 (메모리 누수 등)

---

## 3.3.4 PCB (Process Control Block)

### PCB란?
운영체제가 **프로세스를 관리하기 위해 만든 "신분증"**이다. 각 프로세스마다 하나씩 존재한다.

### PCB에 저장되는 정보

```
┌─────────────────────────────┐
│         PCB 구조            │
├─────────────────────────────┤
│ 1. 프로세스 ID (PID)        │  → 주민등록번호
│ 2. 프로세스 상태             │  → 현재 뭐 하고 있음?
│ 3. 프로그램 카운터 (PC)      │  → 다음에 실행할 명령어 주소
│ 4. CPU 레지스터 정보         │  → 작업하다 잠깐 쉴 때 메모
│ 5. CPU 스케줄링 정보         │  → 우선순위, 큐 포인터
│ 6. 메모리 관리 정보          │  → 코드/데이터/스택 위치
│ 7. 입출력 상태 정보          │  → 사용 중인 파일, 장치
│ 8. 계정 정보                │  → 누가 실행했는지
└─────────────────────────────┘
```

> **컨텍스트 스위칭**: CPU가 다른 프로세스로 전환할 때, 현재 PCB를 저장하고 새 PCB를 불러오는 과정. 이 overhead(비용)가 멀티태스킹의 핵심 비용이다.

---

## 3.3.5 멀티프로세싱 (Multiprocessing)

### 정의
**여러 개의 프로세스를 동시에 실행**하는 것. (진짜 동시는 아니고, CPU가 빠르게 번갈아가며 실행)

### 특징
- 각 프로세스는 **독립된 메모리 공간**을 가짐 (Code, Data, Heap, Stack 모두 별도)
- 한 프로세스가 죽어도 다른 프로세스는 **영향을 받지 않음** (안전!)
- 프로세스 간 통신은 **IPC** (Inter-Process Communication) 필요: 파이프, 소켓, 공유메모리, 메시지 큐 등

### 비유
> **독립된 주방(프로세스) 여러 개**를 운영. 각 주방끼리는 벽으로 막혀 있어서, 한 주방에서 불이 나도 다른 주방은 안전하다. 다만 서로 대화하려면 창문(IPC)을 통해 해야 한다.

---

## 3.3.6 스레드와 멀티스레딩

### 스레드란?
**프로세스 안에서 실행되는 "작업 단위"**. 프로세스는 최소 1개의 스레드를 가진다.

### 프로세스 vs 스레드 메모리 구조

```
[프로세스 A]                    [프로세스 B]
┌─────────────┐               ┌─────────────┐
│  코드       │               │  코드       │
│  데이터     │               │  데이터     │
│  힙        │               │  힙        │
│ ┌────────┐ │               │ ┌────────┐ │
│ │스레드1 │ │               │ │스레드1 │ │
│ │(스택)  │ │               │ │(스택)  │ │
│ ├────────┤ │               │ └────────┘ │
│ │스레드2 │ │               └─────────────┘
│ │(스택)  │ │  ← 스레드끼리는 코드/데이터/힙 공유!
│ └────────┘ │
└─────────────┘
```

| 구분 | 프로세스 | 스레드 |
|------|---------|--------|
| 메모리 공간 | 독립 (전체 별도) | 코드/데이터/힙 **공유**, 스택만 **별도** |
| 통신 방식 | IPC 필요 (복잡) | 전역 변수/힙으로 직접 공유 (간단) |
| 생성 비용 | 크다 (메모리 전체 할당) | 작다 (스택만 추가) |
| 안전성 | 높음 (격리되어 있음) | 낮음 (메모리 공유로 인한 충돌 위험) |
| 죽었을 때 | 다른 프로세스에 영향 없음 | 전체 프로세스가 죽을 수 있음 |

### 멀티스레딩의 장점
- **응답성**: UI 스레드는 화면 그리고, 백그라운드 스레드는 데이터 로딩
- **자원 공유**: 메모리 공유로 데이터 교환이 빠름
- **경제성**: 프로세스 생성보다 스레드 생성이 훨씬 빠름
- **확장성**: 멀티코어 CPU에서 병렬 실행 가능

### 비유
> 프로세스 = **레스토랑(가게)**
> 스레드 = **가게 안의 요리사들**
> 
> 요리사들은 같은 주방(코드/데이터/힙)을 공유하지만, 각자의 작업대(스택)는 따로 쓴다. 협력하면 더 빠르게 일할 수 있지만, 서로 방해하면(경쟁조건) 주문이 꼬일 수 있다.

## 3.3.7 공유 자원과 임계 영역

### 공유 자원 (Shared Resource)
여러 스레드/프로세스가 **함께 사용하는 자원** (변수, 파일, 메모리 등)

### 경쟁 조건 (Race Condition)
여러 스레드가 **동시에** 공유 자원에 접근할 때, 실행 순서에 따라 **결과가 달라지는 문제**

```
예: 공유 변수 count = 0

스레드 A: count = count + 1   (0 → 1 기대)
스레드 B: count = count + 1   (1 → 2 기대)

하지만 동시에 실행되면:
A가 읽음(0) → B가 읽음(0) → A가 씀(1) → B가 씀(1)
결과: count = 1 (기대값 2, 버그!)
```

### 임계 영역 (Critical Section)
**공유 자원에 접근하는 코드 구간**. 이 영역은 한 번에 **하나의 스레드만** 들어가야 한다.

### 상호 배제 (Mutual Exclusion) 해결책

| 방법 | 설명 | 특징 |
|------|------|------|
| **뮤텍스 (Mutex)** | 한 번에 하나만 잠금. 잠금/해제는 같은 스레드가 | 가장 일반적 |
| **세마포어 (Semaphore)** | 정해진 개수(N)만큼 동시 접근 허용 | N=1이면 뮤텍스와 동일 |
| **모니터 (Monitor)** | 고급 언어(Java 등)에서 제공. 뮤텍스 + 조건 변수 | 사용이 편리 |

### 비유
> 임계 영역 = **화장실**
> 
> 화장실(공유 자원)은 한 사람(스레드)만 사용해야 한다. 문 잠금(뮤텍스 lock)을 하고 들어가서 일을 마치면 문을 연다(unlock). 다른 사람은 기다렸다가 들어간다.

## 3.3.8 교착 상태 (Deadlock)

### 정의
두 개 이상의 프로세스/스레드가 **서로의 자원을 기다리며 무한 대기**하는 상태. 아무도 진행하지 못함.

### 교착 상태 4대 조건 (모두 만족해야 발생)

| 조건 | 설명 | 깨는 방법 |
|------|------|----------|
| **상호 배제** | 자원은 한 번에 하나만 사용 | 불가능 (자원 특성) |
| **점유 대기** | 자원을 가진 채로 다른 자원 요청 | 자원을 모두 한 번에 요청 |
| **비선점** | 강제로 빼앗을 수 없음 | 우선순위 높은 프로세스가 강제 점유 |
| **환형 대기** | 원형으로 서로의 자원을 기다림 | 자원에 번호를 매기고 순서대로 요청 |

### 교착 상태 예시 (고전적인 은행가 문제)

```
프로세스 A: 자원 1을 가짐 → 자원 2를 요청 (대기 중)
프로세스 B: 자원 2를 가짐 → 자원 1을 요청 (대기 중)

A는 B가 놓아주길, B는 A가 놓아주길 기다림 → 영원히 대기 (Deadlock!)
```

### 비유
> **좁은 다리에서 두 사람이 마주 보고 서로 비켜주려 함**
> 
> A: "당신 먼저 가세요" → B: "아니 당신 먼저" → 서로 양보 → 영원히 서 있음

### 해결 방법
1. **예방**: 4대 조건 중 하나를 무력화
2. **회피**: 은행가 알고리즘 (안전 상태인지 미리 검사)
3. **탐지 & 회복**: 주기적으로 교착 상태 탐지, 강제 종료
4. **무시**: 발생 확률이 낮으면 그냥 두고 재부팅 (Windows, Linux 등 실용적 접근)

# 3.4 CPU 스케줄링 알고리즘

> **한 줄 요약**: 준비 큐에 있는 많은 프로세스 중, **다음에 누구에게 CPU를 줄까?**를 결정하는 알고리즘

---

## 3.4.1 비선점형 방식 (Non-preemptive)

한 번 CPU를 할당받으면 **끝까지 쓰고 반납**한다. (강제로 빼앗지 않음)

### 1. FCFS (First-Come First-Served)
- **설명**: 먼저 온 순서대로 처리 (FIFO)
- **장점**: 구현이 매우 간단
- **단점**: **Convoy Effect** (짧은 작업이 긴 작업 뒤에서 오래 기다림)

```
예: P1(24ms), P2(3ms), P3(3ms) 이 순서로 도착

간트 차트: | P1(24) | P2(3) | P3(3) |
대기 시간: P1=0, P2=24, P3=27 → 평균 = 17ms

만약 P2, P3이 먼저 왔다면 평균 대기 시간 = 3ms (훨씬 짧음!)
```

### 2. SJF (Shortest Job First)
- **설명**: **실행 시간이 가장 짧은** 프로세스를 먼저
- **장점**: 평균 대기 시간이 **최소**
- **단점**: 긴 작업은 **기아(Starvation)** 현상 발생 (계속 짧은 작업이 먼저)
- **문제**: 실행 시간을 미리 알 수 없음 (예측 필요)

```
예: P1(6), P2(8), P3(7), P4(3)

간트 차트: | P4(3) | P1(6) | P3(7) | P2(8) |
평균 대기 시간 = (0 + 3 + 9 + 16) / 4 = 7ms
FCFS보다 훨씬 짧음!
```

### 3. 우선순위 (Priority)
- **설명**: 우선순위가 높은 프로세스를 먼저
- **단점**: 낮은 우선순위 프로세스 기아 현상
- **해결**: **에이징(Aging)** - 오래 기다린 프로세스의 우선순위를 점진적으로 높임

### 4. HRN (Highest Response-ratio Next)
- **설명**: 대기 시간 + 실행 시간을 고려한 **응답률**이 높은 순서
- **응답률 공식**: `(대기시간 + 실행시간) / 실행시간`
- **장점**: SJF의 기아 문제를 완화 (오래 기다리면 응답률이 높아짐)

---

## 3.4.2 선점형 방식 (Preemptive)

CPU를 **강제로 빼앗을 수 있다**. (더 급한 프로세스가 오면 양보)

### 1. 라운드 로빈 (Round Robin, RR)
- **설명**: 각 프로세스에 **동일한 시간 할당량(Time Quantum)**을 주고 돌아가며 실행
- **장점**: 공평함, 응답 시간이 짧음 (대화형 시스템에 적합)
- **단점**: Time Quantum이 너무 크면 FCFS와 같음, 너무 작으면 컨텍스트 스위칭 overhead 증가

```
예: Time Quantum = 4ms, P1(10), P2(3), P3(5)

간트 차트: | P1(4) | P2(3) | P3(4) | P1(4) | P3(1) | P1(2) |
           0       4       7      11      15      16      18

P1: 10ms, P2: 3ms, P3: 5ms 실행
```

### 2. SRTF (Shortest Remaining Time First)
- **설명**: SJF의 선점형 버전. **남은 실행 시간이 가장 짧은** 프로세스에게 CPU
- **장점**: 평균 대기 시간이 이론적으로 최소
- **단점**: 남은 시간을 계속 추적해야 함, 기아 현상

```
예: P1(8ms, 0초 도착), P2(4ms, 1초 도착), P3(2ms, 2초 도착)

시간 0: P1 실행
시간 1: P2 도착 (P1 남은 7 vs P2 4) → P2 선점!
시간 2: P3 도착 (P2 남은 3 vs P3 2) → P3 선점!
시간 4: P3 완료 → P2 실행
시간 7: P2 완료 → P1 실행
```

### 3. 다단계 큐 (Multilevel Queue)
- **설명**: 프로세스를 여러 그룹(우선순위 큐)으로 나누고, 상위 큐가 우선
- **예시**: 시스템 프로세스(상위) > 대화형 프로세스 > 배치 프로세스(하위)
- **단점**: 한 번 배정된 큐는 이동 불가 → 기아 가능

### 4. 다단계 피드백 큐 (Multilevel Feedback Queue)
- **설명**: 다단계 큐 + **큐 간 이동 가능**. Time Quantum을 다 쓰면 하위 큐로 강등
- **장점**: CPU 집중 프로세스는 하위 큐로, I/O 집중(짧은) 프로세스는 상위 큐로 → 적응적
- **특징**: 현대 OS(Linux, Windows)에서 가장 많이 사용하는 형태

```
┌─────────────────┐  Time Quantum = 8ms
│   Q0 (상위)     │  → 못 끝내면 Q1로
├─────────────────┤  Time Quantum = 16ms
│   Q1            │  → 못 끝내면 Q2로
├─────────────────┤  FCFS (Time Quantum 무한)
│   Q2 (하위)     │  → 여기서도 못 끝내면 계속 대기
└─────────────────┘
```

## 스케줄링 알고리즘 비교 요약

| 알고리즘 | 유형 | 장점 | 단점 | 사용처 |
|---------|------|------|------|--------|
| **FCFS** | 비선점 | 간단함 | Convoy Effect | 배치 시스템 |
| **SJF** | 비선점 | 평균 대기 최소 | 기아, 예측 어려움 | 이론적 최적 |
| **우선순위** | 비선점 | 중요 작업 우선 | 기아 | 실시간 시스템 |
| **HRN** | 비선점 | 기아 완화 | 복잡함 | 배치 시스템 |
| **Round Robin** | 선점 | 공평, 응답 빠름 | Quantum 설정 중요 | 시분할 시스템 |
| **SRTF** | 선점 | 평균 대기 최소 | 구현 복잡, 기아 | 이론적 최적 |
| **다단계 피드백 큐** | 선점 | 적응적, 효율적 | 설정 파라미터 많음 | **현대 OS** |

## 핵심 정리

```
┌─────────────────────────────────────────────┐
│  프로세스 = 독립된 가게 (메모리 전체 별도)    │
│  스레드   = 가게 안의 요리사 (주방 공유)      │
│  PCB     = 프로세스의 신분증                  │
│  임계영역 = 화장실 (한 명씩만!)               │
│  교착상태 = 다리에서 마주보고 양보하는 둘      │
│  스케줄링 = 다음에 누구한테 CPU 줄까?         │
└─────────────────────────────────────────────┘
```

> **비선점형** = 한 번 주면 끝까지 씀 (예의 바른)
> **선점형** = 더 급한 사람 오면 양보 (효율적인)
