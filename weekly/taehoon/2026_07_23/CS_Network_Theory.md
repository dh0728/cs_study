# CS (컴퓨터사이언스 이론) — 네트워크 계층 정리

---

## 1. TCP/IP 4계층 모델

### 📐 계층 구조

| 계층 | 이름 | 주요 프로토콜 | 핵심 역할 |
|------|------|---------------|-----------|
| **4계층** | 응용 계층 (Application) | HTTP, FTP, SMTP, DNS, SSH | 사용자와 직접 상호작용하는 서비스 제공 |
| **3계층** | 전송 계층 (Transport) | TCP, UDP | 종단간(End-to-End) 데이터 신뢰성 보장 및 흐름 제어 |
| **2계층** | 인터넷 계층 (Internet) | IP, ICMP, ARP, IGMP | 논리적 주소(IP)를 이용한 경로 설정(라우팅) |
| **1계층** | 네트워크 인터페이스 계층 (Network Interface) | Ethernet, Wi-Fi, PPP | 물리적 매체를 통한 실제 데이터 전송 |

> **OSI 7계층 vs TCP/IP 4계층 비교**
> - TCP/IP는 OSI의 **세션/표현/응용**을 하나로 묶고, **데이터링크/물리**도 하나로 통합했습니다.
> - TCP/IP는 실제 인터넷 표준이므로, 실무에서는 TCP/IP 모델이 더 많이 쓰입니다.

---

### 📦 PDU (Protocol Data Unit)

> **PDU란?** 각 계층에서 데이터를 부르는 **단위(이름)**입니다. 데이터가 계층을 내려갈 때마다 헤더가 추가되며, 이름도 바뀝니다.

| 계층 | PDU 이름 | 설명 |
|------|----------|------|
| 응용 계층 | **데이터 (Data)** | 사용자가 실제 보내는 메시지, 파일 등 |
| 전송 계층 | **세그먼트 (Segment)** | TCP 사용 시 / **데이터그램 (Datagram)** — UDP 사용 시 |
| 인터넷 계층 | **패킷 (Packet)** | IP 주소가 붙은 데이터 단위 |
| 네트워크 인터페이스 | **프레임 (Frame)** | MAC 주소가 붙은 물리적 전송 단위 |

```
┌─────────────────────────────────────────────┐
│  응용 계층:  "안녕하세요" (데이터)            │
├─────────────────────────────────────────────┤
│  전송 계층:  + TCP 헤더 → 세그먼트            │
├─────────────────────────────────────────────┤
│  인터넷 계층:  + IP 헤더 → 패킷              │
├─────────────────────────────────────────────┤
│  네트워크 인터페이스:  + MAC 헤더 → 프레임     │
└─────────────────────────────────────────────┘
```

## 2. IP 주소

### 🔗 ARP (Address Resolution Protocol)

> **ARP란?** IP 주소를 **MAC 주소(물리적 주소)**로 변환하는 프로토콜

**왜 필요한가?**
- IP 주소는 논리적 주소 → **라우팅**에 사용
- MAC 주소는 물리적 주소 → **실제 LAN 내 전송**에 사용
- 같은 네트워크(서브넷) 내에서 통신할 때는 **MAC 주소가 필수**

**ARP 동작 과정:**

```
┌─────────────────────────────────────────────────────────┐
│  1. PC-A가 PC-B에게 데이터를 보내고 싶음                  │
│     → PC-A는 PC-B의 IP 주소는 알지만 MAC 주소는 모름      │
│                                                         │
│  2. PC-A가 LAN 전체에 "ARP 요청" 브로드캐스트 전송        │
│     "IP 192.168.1.5를 가진 장치, 너의 MAC 주소 알려줘!"  │
│                                                         │
│  3. PC-B가 자신의 MAC 주소를 담아 "ARP 응답" 유니캐스트   │
│     "나야! 내 MAC은 AA:BB:CC:DD:EE:FF야"                 │
│                                                         │
│  4. PC-A가 ARP 캐시 테이블에 저장 → 이후 통신 시 재사용   │
└─────────────────────────────────────────────────────────┘
```

| 항목 | 설명 |
|------|------|
| **ARP 캐시** | IP-MAC 매핑을 임시 저장하는 테이블 (시간이 지나면 삭제) |
| **RARP** | 반대로 MAC → IP를 알아내는 프로토콜 (오늘날 거의 사용 안 함) |
| **Proxy ARP** | 라우터가 대신 ARP 응답을 해주는 방식 |

### 🚀 홉바이홉 (Hop-by-Hop) 통신

> **홉(Hop)이란?** 데이터가 네트워크를 이동할 때 거치는 **각 라우터(중계 장치)** 하나를 의미

**핵심 개념:**
- 데이터는 출발지에서 목적지까지 **직접 가지 않고**, 여러 라우터를 **홉 단위로 점프**하며 이동
- 각 라우터는 **자신의 다음 홉(Next Hop)**만 알고 있음

```
출발지 PC ──[홉1]── 라우터A ──[홉2]── 라우터B ──[홉3]── 라우터C ── 목적지 서버
              ↑           ↑           ↑
           각 라우터는 자신의 다음 홉만 결정 (라우팅 테이블 참고)
```

**라우팅 테이블 예시:**

| 목적지 네트워크 | 서브넷 마스크 | 다음 홉(게이트웨이) | 인터페이스 |
|----------------|--------------|-------------------|-----------|
| 192.168.1.0 | 255.255.255.0 | 직접 연결 | eth0 |
| 10.0.0.0 | 255.0.0.0 | 192.168.1.1 | eth0 |
| 0.0.0.0 (기본) | 0.0.0.0 | 192.168.1.1 | eth0 |

> **TTL (Time To Live):** 패킷이 무한 루프에 빠지지 않도록, 각 홉을 지날 때마다 1씩 감소. 0이 되면 폐기.

### 🏠 IP 주소 체계

#### IPv4 vs IPv6

| 구분 | IPv4 | IPv6 |
|------|------|------|
| **주소 길이** | 32비트 (약 43억 개) | 128비트 (약 3.4×10³⁸ 개) |
| **표현 방식** | 8비트 × 4 = 4옥텟 (ex: 192.168.1.1) | 16비트 × 8 = 8그룹 (ex: 2001:0db8::1) |
| **헤더 길이** | 가변 (20~60바이트) | 고정 (40바이트) |
| **보안** | 옵션 (IPsec) | 기본 내장 (IPsec) |
| **NAT 필요** | 주소 부족으로 필수 | 불필요 (주소 충분) |

#### IPv4 클래스 기반 주소 체계 (Classful)

| 클래스 | 첫 옥텟 범위 | 기본 서브넷 마스크 | 용도 | 예시 |
|--------|-------------|-------------------|------|------|
| **A** | 0~127 | 255.0.0.0 (/8) | 대규모 네트워크 | 10.0.0.0 ~ 10.255.255.255 |
| **B** | 128~191 | 255.255.0.0 (/16) | 중간 규모 | 172.16.0.0 ~ 172.31.255.255 |
| **C** | 192~223 | 255.255.255.0 (/24) | 소규모 네트워크 | 192.168.0.0 ~ 192.168.255.255 |
| **D** | 224~239 | - | 멀티캐스트 | 224.0.0.0 ~ 239.255.255.255 |
| **E** | 240~255 | - | 연구/예약 | 240.0.0.0 ~ 255.255.255.255 |

#### 사설 IP 주소 (Private IP)

> 인터넷에 직접 노출되지 않는, 내부 네트워크 전용 주소

| 클래스 | 사설 IP 범위 |
|--------|-------------|
| A | 10.0.0.0 ~ 10.255.255.255 |
| B | 172.16.0.0 ~ 172.31.255.255 |
| C | 192.168.0.0 ~ 192.168.255.255 |

> 사설 IP는 **NAT**을 통해 공인 IP로 변환되어 인터넷과 통신합니다.

#### CIDR (Classless Inter-Domain Routing)

> 클래스 개념을 버리고, **서브넷 마스크 길이를 자유롭게** 지정하는 방식

| 표기 | 의미 | 사용 가능 IP 수 |
|------|------|----------------|
| 192.168.1.0/24 | 서브넷 마스크: 255.255.255.0 | 256개 (2⁸) |
| 192.168.1.0/26 | 서브넷 마스크: 255.255.255.192 | 64개 (2⁶) |
| 10.0.0.0/8 | 서브넷 마스크: 255.0.0.0 | 16,777,216개 (2²⁴) |

**/24와 /26 비교:**

```
192.168.1.0/24  →  네트워크: 192.168.1.0, 호스트: 1~254 (256개)
192.168.1.0/26  →  네트워크: 192.168.1.0, 호스트: 1~62 (64개)
                    네트워크: 192.168.1.64, 호스트: 65~126
                    네트워크: 192.168.1.128, 호스트: 129~190
                    네트워크: 192.168.1.192, 호스트: 193~254
```

### 📍 IP 주소를 이용한 위치 정보

#### IP 주소와 지리적 위치

| 특징 | 설명 |
|------|------|
| **원리** | IP 주소는 ISP(인터넷 서비스 제공자)가 **지역별로 블록 단위로 할당** |
| **정확도** | 국가/도시 수준은 높음, **건물/집 주소 수준은 낮음** |
| **동적 IP** | 가정용 인터넷은 대부분 동적 IP → 위치가 자주 바뀜 |
| **VPN/프록시** | IP 위치를 숨기거나 위장할 수 있음 |

**IP 위치 추적 과정:**

```
사용자 IP (예: 203.XX.XX.XX)
        ↓
    WHOIS 데이터베이스 조회
        ↓
    ISP 정보 확인 (예: KT, SK브로드밴드)
        ↓
    ISP의 IP 할당 블록에서 지역 확인
        ↓
    대략적인 위치: "서울특별시 강남구" (정확도: 수 km)
```

#### 실제 활용 사례

| 서비스 | 활용 방식 |
|--------|-----------|
| **지도 서비스** | "현재 위치" 기능 (브라우저 IP 기반) |
| **콘텐츠 지역 제한** | Netflix, YouTube — IP 기반 국가 제한 |
| **보안/부정 로그인 감지** | 은행, 게임 — 비정상 지역 로그인 시 알림 |
| **광고 타겟팅** | 지역 기반 광고 노출 |

#### 한계점

> ⚠️ **IP 주소만으로 정확한 실제 위치(집 주소)를 알 수 없습니다.**

- 동일 IP를 수천 명이 공유 (공유기, 모바일 통신사)
- VPN 사용 시 완전히 다른 국가 IP로 표시
- IP 위치 DB는 ISP 등록 정보 기반이라 실시간성 부족

## 📋 핵심 요약

| 주제 | 핵심 정리 |
|------|----------|
| **TCP/IP 4계층** | 응용 → 전송 → 인터넷 → 네트워크 인터페이스 |
| **PDU** | 데이터 → 세그먼트 → 패킷 → 프레임 |
| **ARP** | IP 주소 → MAC 주소 변환 (같은 네트워크 내 필수) |
| **홉바이홉** | 데이터는 라우터를 거쳐 점프하며 이동 |
| **IP 주소 체계** | IPv4(32bit) / IPv6(128bit), 클래스 → CIDR |
| **IP 위치** | 국가/도시 수준 추적 가능, 정확한 주소는 불가 |
