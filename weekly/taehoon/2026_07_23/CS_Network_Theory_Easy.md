## 1. TCP/IP 4계층 모델

### 📐 계층 구조 = "택배 배송 과정"으로 이해하기

택배가 집에서 집으로 가는 과정을 생각해보세요. TCP/IP도 똑같아요!

```
┌─────────────────────────────────────────────────────────────┐
│  🏠 4계층: 응용 계층 (Application)                          │
│     → "택배 상자 안에 넣을 물건"                            │
│     예: 웹사이트 보기(HTTP), 이메일 보내기(SMTP), 파일 전송 │
│     → 사용자가 직접 만지는 부분!                            │
├─────────────────────────────────────────────────────────────┤
│  🚚 3계층: 전송 계층 (Transport)                              │
│     → "택배 상자 + 운송장"                                  │
│     예: TCP(안전한 배송), UDP(빠른 배송)                     │
│     → 물건이 제대로 도착했는지 확인, 잃어버리면 다시 보내기    │
├─────────────────────────────────────────────────────────────┤
│  🗺️ 2계층: 인터넷 계층 (Internet)                            │
│     → "우편번호 + 주소"                                     │
│     예: IP 주소                                             │
│     → 어디로 보낼지 경로 찾기 (라우팅)                       │
├─────────────────────────────────────────────────────────────┤
│  🚛 1계층: 네트워크 인터페이스 계층 (Network Interface)        │
│     → "실제 트럭 + 도로"                                    │
│     예: Ethernet 케이블, Wi-Fi                              │
│     → 전기 신호로 실제로 데이터를 보내는 물리적 부분         │
└─────────────────────────────────────────────────────────────┘
```

> **한 문장 정리**: 데이터는 위에서 아래로 내려가며 **포장(헤더 추가)**되고, 도착하면 아래에서 위로 **뜯어(헤더 제거)** 읽힙니다.

---

### 📦 PDU (Protocol Data Unit) = "데이터 이름이 계층마다 달라요"

| 계층 | PDU 이름 | 비유 | 설명 |
|------|----------|------|------|
| 응용 | **데이터** | 편지지 본문 | 사용자가 쓴 실제 내용 |
| 전송 | **세그먼트** | 봉투 + 우편번호 | TCP 헤더(포트 번호) 추가 |
| 인터넷 | **패킷** | 택배 상자 + 주소 라벨 | IP 헤더(발신/수신 IP) 추가 |
| 네트워크 인터페이스 | **프레임** | 트럭에 실은 화물 | MAC 주소 + 전기 신호 변환 |

```
📧 편지 쓰기 (응용 계층: 데이터)
    ↓ + 봉투에 넣기 (전송 계층: 세그먼트)
    ↓ + 택배 상자 포장 (인터넷 계층: 패킷)
    ↓ + 트럭에 실기 (네트워크 인터페이스: 프레임)
    ↓
    🚛 실제 도로로 이동!

    (도착하면 반대로 뜯어 읽음)
```

## 2. IP 주소

### 🔗 ARP = "IP 주소로 MAC 주소를 찾아내는 전화번호부"

#### 왜 ARP가 필요할까?

| 주소 종류 | 비유 | 역할 |
|-----------|------|------|
| **IP 주소** | 집 주소 (논리적) | "서울시 강남구 테헤란로 123" — 큰 범위에서 찾기 |
| **MAC 주소** | 주민등록번호 (물리적) | "홍길동, 000000-0000000" — 정확히 누구인지 식별 |

> **같은 아파트(네트워크) 안에서 통신할 때는 주민등록번호(MAC)가 필요해요!**

#### ARP 동작 과정 = "아파트 엘리베이터에서 이름 부르기"

```
🏢 같은 아파트(네트워크)에 사는 사람들

PC-A: "야, 302호(IP 192.168.1.5) 사는 사람!"
      → 엘리베이터 안에서 소리침 (브로드캐스트)

      ┌────────────────────────────┐
      │  101호: "나 아닌데?"        │
      │  202호: "나도 아닌데?"      │
      │  302호(PC-B): "여기야!"    │
      └────────────────────────────┘

PC-B: "나 302호야, 내 주민번호(MAC)는 AA:BB:CC:DD:EE:FF야"
      → PC-A에게만 귓속말 (유니캐스트)

PC-A: 📝 메모장에 "302호 = AA:BB:CC:DD:EE:FF" 기록
      → 다음에 또 찾을 때 메모장 보고 바로 찾음 (ARP 캐시)
```

| 용어 | 설명 |
|------|------|
| **ARP 요청** | "누구야?" — 모두에게 물어봄 |
| **ARP 응답** | "나야!" — 해당 장치만 답함 |
| **ARP 캐시** | 메모장. 5~20분 후 지워짐 |
| **RARP** | 반대로 주민번호로 집 주소 찾기 (요즘 안 씀) |

---

### 🚀 홉바이홉 (Hop-by-Hop) 통신 = "택배가 물류센터를 거쳐 가는 것"

#### 홉(Hop)이 뭔가요?

> **홉 = 데이터가 지나가는 라우터(물류센터) 하나**

```
🚚 택배 배송 과정

서울(출발지) 
    ↓ [홉1] 경기 물류센터(라우터A)
    ↓ [홉2] 대전 물류센터(라우터B)  
    ↓ [홉3] 부산 물류센터(라우터C)
    ↓
    🏠 부산(목적지)

    → 각 물류센터는 "다음 물류센터"만 알고 있음!
    → 서울 물류센터는 부산 물류센터 위치를 몰라도 됨
```

#### 라우팅 테이블 = "물류센터의 배송 지도"

| 목적지 | 다음에 갈 곳 | 비유 |
|--------|-------------|------|
| 같은 동네 | 직접 배달 | 같은 아파트면 문 앞까지 |
| 다른 동네 | OO 물류센터로 보냄 | 다른 동네면 물류센터로 |
| 모르는 곳 | 기본 물류센터로 보냄 | 기본 게이트웨이 |

> **TTL (Time To Live)**: 택배가 무한히 돌아다니지 않게, 물류센터를 지날 때마다 스탬프 찍음. 64번 지나면 버림!

### 🏠 IP 주소 체계 = "집 주소 체계"

#### IPv4 vs IPv6 = "4자리 우편번호 vs 8자리 우편번호"

| | IPv4 | IPv6 |
|--|------|------|
| **주소 길이** | 32비트 (4자리) | 128비트 (8자리) |
| **주소 개수** | 약 43억 개 | 약 340涧개 (거의 무한) |
| **예시** | `192.168.1.1` | `2001:0db8:85a3::1` |
| **문제** | 사람이 너무 많아서 주소 부족! | 주소 충분, 미래형 |

> **IPv4 주소 부족 문제** = 서울 아파트가 43억 채밖에 없는데, 세계 인구가 80억...  
> → **NAT**으로 여러 집이 같은 공인 IP를 공유하는 꼼수 사용

#### 사설 IP vs 공인 IP = "아파트 호수 vs 도로명 주소"

| | 사설 IP | 공인 IP |
|--|---------|---------|
| **비유** | 아파트 호수 (101호, 202호) | 실제 도로명 주소 |
| **범위** | 192.168.x.x, 10.x.x.x 등 | ISP가 할당한 유일한 주소 |
| **인터넷 접속** | ❌ 직접 불가 | ✅ 직접 가능 |
| **보안** | 외부에서 직접 접근 불가 | 외부에서 접근 가능 |

```
        🏠 집 (사설 IP: 192.168.1.5)
            ↓
        📡 공유기 (NAT 변환)
            ↓
        🌐 인터넷 (공인 IP: 203.123.45.67)

    → 집 안 모든 기기가 공유기를 통해 하나의 공인 IP로 나감
    → 외부에서는 "203.123.45.67" 하나만 보임
```

#### CIDR = "우편번호를 유연하게 쓰기"

> 원래는 A/B/C 클래스로 고정 → 이제는 **원하는 대로 자르기**

| 표기 | 의미 | 사용 가능 | 비유 |
|------|------|-----------|------|
| `/24` | 24비트가 네트워크 | 256개 | 아파트 1동 (256세대) |
| `/26` | 26비트가 네트워크 | 64개 | 아파트 1동을 4개로 쪼갬 |
| `/8` | 8비트가 네트워크 | 1,600만 개 | 도시 전체 |

```
192.168.1.0/24  →  1동 전체 (256세대)
    ↓ 쪼개기
    192.168.1.0/26   →  1~64호 (64세대)
    192.168.1.64/26  →  65~128호
    192.168.1.128/26 →  129~192호
    192.168.1.192/26 →  193~256호
```

### 📍 IP 주소를 이용한 위치 정보 = "우편번호로 대략적인 지역 알아내기"

#### 원리 = "우편번호는 지역별로 배정됨"

| 정보 | 설명 |
|------|------|
| **IP 할당** | KT는 서울에, SK는 부산에 블록 단위로 IP 줌 |
| **정확도** | "서울시 강남구" 수준은 맞음 |
| **정확도 한계** | "강남대로 123번지 45호"는 못 알아냄 |

#### IP 위치 추정 과정

```
사용자 IP: 203.123.45.67
    ↓
    WHOIS 조회: "이 IP는 KT가 관리함"
    ↓
    KT의 IP 할당 기록: "203.123.0.0 ~ 203.123.255.255 = 서울 지역"
    ↓
    결과: "서울특별시" (정확도: 수 km)
```

#### 실제 활용 vs 한계

| 활용 | 예시 |
|------|------|
| ✅ 넷플릭스 | "한국 콘텐츠만 보여줄게" |
| ✅ 은행 보안 | "평소 서울에서 로그인하다가 갑자기 해외?" → 알림 |
| ✅ 광고 | "강남구 주민에게 맛집 광고" |
| ❌ 정확한 주소 | IP만으로는 집 주소 못 찾음 |
| ❌ VPN 우회 | VPN 쓰면 "미국 IP"로 나와서 속임 |

> ⚠️ **중요**: IP 주소만으로는 **정확한 실제 위치(집 주소)를 알 수 없습니다.**  
> 수천 명이 같은 공유기 IP를 쓰고, VPN으로 위장도 가능하니까요!

## 📋 최종 요약 — 한 눈에 보기

| 개념 | 비유 | 핵심 |
|------|------|------|
| **TCP/IP 4계층** | 택배 배송 과정 | 응용(물건) → 전송(포장) → 인터넷(주소) → 인터페이스(도로) |
| **PDU** | 포장 단계 이름 | 데이터 → 세그먼트 → 패킷 → 프레임 |
| **ARP** | 아파트에서 이름 부르기 | IP 주소로 MAC 주소 찾기 |
| **홉바이홉** | 물류센터 거쳐가기 | 라우터를 하나씩 점프하며 이동 |
| **IP 주소** | 집 주소 체계 | IPv4(부족) → IPv6(충분), 사설/공인, NAT |
| **IP 위치** | 우편번호로 지역 추정 | 국가/도시 수준 가능, 정확한 주소는 불가 |
