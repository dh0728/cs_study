# 자료구조-복잡도

# 복잡도(Complexity)

> <b>💡 한 줄 요약</b>
> 알고리즘이 얼마나 효율적이고 똑똑하게 작동하는지 측정하는 객관적 기준인 <b>시간 복잡도</b>와 <b>공간 복잡도</b>를 마스터하고, 자원 간의 <b>트레이드오프(Trade-off)</b>를 이해하는 단원입니다.

---

## 1. 복잡도(Complexity) 개요

작성한 코드가 얼마나 효율적이고 똑똑하게 작동하는지 평가하기 위해 <b>시간 복잡도</b>와 <b>공간 복잡도</b>라는 기준을 사용합니다. 좋은 알고리즘이란 <b>실행 시간도 짧으면서 저장 공간(메모리)도 적게 쓰는 알고리즘</b>을 의미합니다. 하지만 일반적으로 시간과 공간은 서로 <b>반비례적인 경향(Trade-off)</b>이 있어 둘을 동시에 완벽히 만족하기는 어렵습니다.

* **과거 환경:** 하드웨어 자원이 부족하여 메모리를 아끼는 공간 복잡도가 매우 중요했습니다.
* **현대 환경:** 대용량 인프라와 하드웨어(RAM)의 비약적인 발전으로 시스템이 보편화되었기 때문에, 프로그램을 구현할 때 공간 복잡도보다는 <b>시간 복잡도가 최우선순위</b>가 되었습니다. 임베디드 시스템이나 대용량 데이터를 다루는 특수한 환경에서는 여전히 메모리 효율이 매우 중요한 요소로 취급됩니다.

---

## 2. 시간 복잡도 (Time Complexity)

### 2.1 시간 복잡도란?
> <b>"문제를 해결하는 데 걸리는 시간과 입력 크기($n$)의 함수 관계"</b>

컴퓨터의 사양, 메모리 성능, 프로그래밍 언어의 종류 등 실행 환경에 따라 실제 실행 시간(초)은 제각각 다르기 때문에 객관적인 기준이 될 수 없습니다. 따라서 시간 복잡도는 실제 초 단위 시간이 아닌 <b>"입력 데이터의 크기에 따라 핵심 로직(명령문)이 몇 번 반복(연산)되는가"</b>를 기준으로 측정합니다. 입력값이 적을 때는 차이가 크지 않지만, 대용량 데이터(큰 입력값)를 다룰 때 알고리즘 간의 성능 차이가 극명하게 드러납니다.

### 2.2 Big-O(빅-오) 표기법
입력 범위 $n$을 기준으로 복잡도에 가장 영향을 많이 끼치는 항의 상수 인자를 빼고 나머지 항을 없애서 효율성을 나타내는 표기법입니다. 빅-오 표기법은 <b>상한 점근(Asymptotic Upper Bound)</b>을 따르므로 발생할 수 있는 <b>최악의 경우(Worst Case)</b>를 기준으로 삼아, 프로그램 실행 과정에서 발생할 수 있는 최악의 소요 시간까지 안전하게 고려합니다.

#### 💡 빅-오 표기 규칙 및 원리
1. <b>상수 및 낮은 차수의 항은 무시한다.</b>
   * 입력 크기 $n$이 무한히 커질수록 최고차항이 전체 실행 시간을 완전히 지배하고, 나머지 항들의 영향은 무시할 수 있을 만큼 작아지기 때문입니다.
2. **예시:**
   * $T(n) = 10n^2 + n = O(n^2)$
   * $T(n) = n^2 + 2n + 1 = O(n^2)$
   * $T(n) = 4n^2 + 3n + 21 \rightarrow O(n^2)$

## 3. 주요 Big-O 표기법의 종류 및 특징

아래는 시간 복잡도의 속도 비교 및 대표적인 자료구조/알고리즘 연산 예시입니다. (<b>※ 암기 필수!</b>)

| 표기법 | 명칭 (Time) | 속도 특징 | 대표 예시 및 자료구조 연산 |
| :--- | :--- | :--- | :--- |
| **$O(1)$** | **상수 시간<br>(Constant)** | 데이터가 1개든 100만 개든 입력 크기와 무관하게 항상 일정한 연산 횟수를 가짐. 가장 이상적인 속도. | 배열의 인덱스 참조 (`arr[i]`), 해시 테이블 탐색/삽입/삭제, 스택/큐의 push/pop |
| **$O(\log n)$** | **로그 시간<br>(Logarithmic)** | 문제를 해결하는 데 필요한 단계들이 연산마다 절반씩 줄어듦. 입력이 2배 증가해도 연산은 1번만 증가하므로 효율성이 매우 높음. | 이진 탐색 (Binary Search), 힙(Heap) 삽입/삭제, 이진 탐색 트리(BST) 탐색 |
| **$O(n)$** | **선형 시간<br>(Linear)** | 입력값 $n$이 증가함에 따라 연산 횟수(시간)도 같은 비율로 직선적으로 정비례하여 증가함. | 정렬되지 않은 리스트에서 최댓값/최솟값 탐색, 연결 리스트 순회 |
| **$O(n \log n)$** | **선형 로그 시간<br>(Linearithmic)** | 데이터의 수가 2배로 늘 때, 연산 횟수는 2배보다 약간 더 많이 증가함. 효율적인 정렬 알고리즘의 한계선. | 병합 정렬 (Merge Sort), 퀵 정렬 (Quick Sort, 평균), 힙 정렬 (Heap Sort) |
| **$O(n^2)$** | **제곱 시간<br>(Quadratic)** | 입력 크기의 제곱에 비례하여 연산이 증가함. 주로 2중 반복문(Nested Loop)에서 발생하며 데이터가 커질수록 성능이 급격히 저하됨. | 버블 정렬 (Bubble Sort), 삽입 정렬 (Insertion Sort), 선택 정렬 (Selection Sort) |
| **$O(2^n)$** | **지수 시간<br>(Exponential)** | 입력이 1 증가할 때마다 연산 횟수가 2배씩 제곱수 형태로 폭발적으로 증가함. 실용적으로 사용하기 힘든 구조. | 피보나치 수열의 단순 재귀(Recursion) 구현, 브루트 포스(모든 경우의 수 탐색) |
| **$O(n!)$** | **팩토리얼 시간<br>(Factorial)** | 빅오 표기법 중 가장 느린 최악의 속도. 데이터가 조금만 늘어나도 컴퓨터가 과부하에 걸려 처리 불가능. | 순열(Permutation)의 모든 경우의 수 도출 |

### 🚀 시간 복잡도 속도 비교 및 시각화 그래프
$$\bf O(1) < O(\log n) < O(n) < O(n \log n) < O(n^2) < O(2^n) < O(n!)$$

## 4. 코드로 보는 시간 복잡도 예시 (Python)

### 4.1 $O(1)$ — 상수 시간 (Constant time)
입력 크기 $n$에 관계없이 항상 고정된 횟수(1번)의 연산만 수행하는 구조입니다.
```python
def get_first_element(arr):
    return arr[0]  # 배열 크기와 무관하게 항상 한 번의 연산
```

### 4.2 $O(\log n)$ — 로그 시간 (Logarithmic time)
입력 크기 $n$이 커져도 연산 횟수가 로그 비율로 증가하는 알고리즘입니다. 매 단계마다 탐색 범위를 절반씩 줄여나가는 방식이 대표적입니다.

```python
def binary_search(arr, target):
    left, right = 0, len(arr) - 1
    while left <= right:
        mid = (left + right) // 2
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            left = mid + 1
        else:
            right = mid - 1
    return -1
```

* **왜 $logN$인가?**
  먼저 로그란 로그는 **"몇 번 곱해야 해?"** 에 대한 답을 말합니다.
  * $2^? = 8 \Rightarrow \log_2 8 = 3$
  * "2를 몇 번 곱해야 8이 돼?" $\rightarrow$ **3번!**

#### 📐 로그 표기법 정리
## $$\log_{\color{red}{밑}} {\color{blue}{진수}} = {\color{green}{지수}}$$
* $\color{red}{\text{밑(2)}}$ : 몇 배씩 커지는지
* $\color{blue}{\text{진수(8)}}$ : 목표 숫자
* $\color{green}{\text{지수(3)}}$ : 몇 번 곱했는지

이제 로그N이란?
* "N개를 2로 몇 번 나눠야 1이 되는가?" 에 대한 답을 말한다.
* $N \div 2^k = 1 \Rightarrow k = \log_2 n$

이제 1~100 사이의 특정 숫자를 찾는 경우

* <b>😅 최악의 경우 - 1부터 하나씩 물어보기:</b>
  ```python
  "1이야?" → 아니 
  "2이야?" → 아니
  "3이야?" → 아니
  ... 최악의 경우 100번 질문 → O(n)
  ```
* <b>😎 이진탐색을 활용할 경우:</b>
  ```python
  "50이야?" → 아니, 더 커!
      → 1~49 는 한번에 버림 🗑️ (50개 제거)
  
  "75이야?" → 아니, 더 작아!
      → 76~100 는 한번에 버림 🗑️ (25개 제거)
  
  "62이야?" → 아니, 더 커!
      → 51~61 는 한번에 버림 🗑️ (12개 제거)
  
  "68이야?" → 정답! 🎉
  ```
  $\rightarrow$ 단 4번만에 원하는 수를 찾음. $n=1,000$이어도 약 10번의 연산으로 탐색이 가능합니다.

### 4.3 $O(n)$ — 선형 시간 (Linear time)
입력 크기 $n$에 비례하여 연산 횟수가 증가하는 알고리즘입니다. 입력 데이터를 처음부터 끝까지 한 번씩 순회하는 경우가 대표적입니다.
```python
def find_max(arr):
    max_val = arr[0]
    for num in arr:  # n번 순회
        if num > max_val:
            max_val = num
    return max_val
```

### 4.4 $O(n^2)$ — 제곱 시간 (Quadratic time)
입력 크기 $n$의 제곱에 비례하여 연산 횟수가 증가하는 알고리즘입니다. 이중 반복문을 사용하는 경우가 대표적이며, $n$이 커질수록 성능이 급격히 저하됩니다.
```python
def bubble_sort(arr):
    n = len(arr)
    for i in range(n):               # 외부 반복문: n번
        for j in range(n - i - 1):   # 내부 반복문: n번
            if arr[j] > arr[j + 1]:
                arr[j], arr[j + 1] = arr[j + 1], arr[j]
    return arr
```

### 4.5 $O(2^n)$ — 지수 시간 (Exponential time)
입력 크기 $n$이 1 증가할 때마다 연산 횟수가 2배씩 증가하는 알고리즘입니다. 재귀적으로 모든 경우의 수를 탐색하는 경우가 대표적이며, $n$이 조금만 커져도 실행 시간이 폭발적으로 늘어납니다.
```python
def fibonacci(n):
    if n <= 1:
        return n
    return fibonacci(n - 1) + fibonacci(n - 2)  # 두 번의 재귀 호출
```
* **설명:** `fibonacci(4)`를 호출했을 때 총 9번이 호출되는 실행 플로우 구조입니다.
  ```python
  1층  :        f(4)         → 1개
  2층  :    f(3)   f(2)      → 2개
  3층  :  f(2) f(1) f(1) f(0) → 4개
  4층  : f(1)f(0)            → 8개 (최대)
  ```
* <b>🔢 호출 횟수가 늘어나는 추이 테이블:</b> 호출 횟수가 $2^n$을 넘진 않지만 거의 비슷하게 증가합니다.

| n | 호출 횟수 | $2^n$ |
| :-: | :-: | :-: |
| 1 | 1 | 2 |
| 2 | 3 | 4 |
| 3 | 5 | 8 |
| 4 | 9 | 16 |
| 5 | 15 | 32 |

## 5. 공간 복잡도 (Space Complexity)

### 5.1 공간 복잡도란?
> <b>"알고리즘을 실행하는 데 필요한 메모리 양"</b>

시간 복잡도와 마찬가지로 <b>Big-O 표기법</b>을 사용하여 나타냅니다.

### 5.2 특징 및 구성 요소
공간 복잡도는 <b>보조공간(Auxiliary Space)</b>과 <b>입력 공간(Input size)</b>을 합친 포괄적인 개념입니다.

* **입력 공간(Input size):** 알고리즘에 전달되는 입력 데이터 자체가 차지하는 메모리 공간 (예: 크기 $n$인 배열).
* **보조공간(Auxiliary Space):** 알고리즘이 실행되는 동안 입력 외에 추가적으로 사용하는 임시 메모리 공간 (예: 정렬 중 생성되는 임시 배열, 변수, 재귀 호출 스택 등).
* **💡 핵심 분석 포인트:** 변수, 상수와 같이 입력 크기와 무관한 고정 공간과 동적 할당, 재귀 스택처럼 입력 크기에 의존하는 가변 공간 중, 실질적인 복잡도 계산 시에는 <b>가변 공간 및 보조 공간</b>에 집중합니다.

### 5.3 공간 복잡도가 필요한 이유?
메모리 자원은 한정되어 있기 때문에, 알고리즘이 얼마나 많은 메모리를 사용하는지 파악하는 것이 중요합니다. 특히 임베디드 시스템이나 대용량 데이터를 다루는 환경에서는 메모리 효율이 성능만큼 중요한 요소가 됩니다. 공간복잡도를 분석하면 시간과 메모리 간의 <b>트레이드오프(trade-off)</b>를 고려한 최적의 알고리즘을 선택할 수 있습니다.

### 5.4 데이터 차원에 따른 공간 복잡도 기준 및 상세 계산법
* 1차원 배열 크기 $n \rightarrow O(n)$
* 2차원 배열 크기 $n \times n \rightarrow O(n^2)$
* 3차원 배열 크기 $n \times n \times n \rightarrow O(n^3)$

#### 📐 파이썬 메모리 기반의 공간 복잡도 계산 수식 증명
```python
def sum_array(a: list, n: int) -> int:
    x = 0
    for i in range(n):
        x = x + a[i]
    return x
```
* `a` (list) : 요소당 28 byte + 요소 크기 $\times n$ <b>(입력 공간)</b>
* `n` (int) : 28 byte <b>(입력 공간)</b>
* `x` (int) : 28 byte <b>(보조 공간)</b>
* `i` (int) : 28 byte <b>(보조 공간)</b>

위 코드의 메모리 소모 총합을 수식으로 정리하면 다음과 같으며, 최종적으로 36n + 140의 메모리를 요구하여 선형적으로 증가하기 때문에 공간복잡도는 **$O(n)$**이 됩니다.
$$\bf \underbrace{(56 + 36n)}_{a} + \underbrace{28}_{n} + \underbrace{28}_{x} + \underbrace{28}_{i} = 36n + (56 + 28 + 28 + 28) = 36n + 140 \text{ (byte)}$$

## 6. 코드로 보는 공간 복잡도 예시 (Python)

### 6.1 $O(1)$ — 상수 공간 복잡도
입력 배열 `arr`의 크기가 아무리 커져도, 데이터를 저장할 추가적인 임시 배열을 만들지 않고 고정된 변수들만 사용하므로 보조 공간은 항상 고정되어 있습니다.
```python
def sum_all(arr):
    total = 0           # 변수 1개만 사용 (보조 공간 O(1))
    for x in arr:
        total += x
    return total
```

### 6.2 $O(n)$ — 선형 공간 복잡도
입력 크기 $n$만큼의 새로운 리스트 `result`를 생성하여 값을 채우므로, 입력값에 비례하여 메모리 사용량이 정비례로 증가합니다.
```python
def double_all(arr):
    result = []         # 입력 크기만큼 새 배열 생성 (가변 공간 생성)
    for x in arr:
        result.append(x * 2)
    return result
```

### 6.3 $O(n^2)$ — 이차 공간 복잡도
행과 열의 크기가 모두 $n$에 영향을 받는 2차원 배열 구조를 생성하므로 $n \times n$ 크기의 메모리 공간을 소모하게 됩니다.
```python
def make_matrix(n):
    matrix = [[0] * n for _ in range(n)]  # n x n 2차원 배열 생성
    return matrix
```

## 7. 🔥 [CS 심화 스터디] 개념 확장 및 이해를 돕는 추가 자료

CS 전공 시험 및 기술 면접에서 고득점을 얻기 위해 반드시 알아야 할 <b>복잡도의 심화 이론과 실무 트레이드오프</b> 내용입니다.

### 7.1 재귀 함수(Recursion)와 콜 스택(Call Stack)의 공간 복잡도 함정
루프(반복문)를 쓰지 않고 재귀를 쓸 때, 화면상에 명시적인 배열을 선언하지 않았더라도 메모리가 터지는 경우가 발생합니다. 함수가 호출될 때마다 시스템 메모리의 <b>콜 스택(Call Stack)</b>에 함수의 복귀 주소와 지역 변수가 세로로 차곡차곡 쌓이기 때문입니다.

#### ❌ 복잡도가 $O(2^n)$ 시간 / $O(n)$ 공간이 되는 지수 재귀 함수 (피보나치 수열)
```python
def fibonacci(n):
    if n <= 1:
        return n
    return fibonacci(n - 1) + fibonacci(n - 2)  # 두 번의 재귀 호출
```
* **시간 복잡도 $O(2^n)$:** `fibonacci(4)`를 호출하면 하위 호출이 `f(4)` $\rightarrow$ `f(3), f(2)` $\rightarrow$ `f(2), f(1), f(1), f(0)` 형태로 트리 구조로 늘어나 총 9번 호출됩니다. 호출 횟수가 $2^n$과 거의 비슷하게 폭발적으로 증가하는 최악의 시간 효율을 보입니다.
* **공간 복잡도 $O(n)$:** 데이터를 보관하는 배열은 없지만, 함수가 실행되면서 밑으로 파고 내려가는 최대 재귀 깊이(Depth)가 $n$에 도달하므로 콜 스택이 $n$층만큼 쌓여 공간 복잡도는 $O(n)$이 됩니다.

### 7.2 트레이드오프(Trade-off) 실무 활용: 메모이제이션(Memoization)
시간과 공간의 트레이드오프 관계를 가장 극적으로 활용한 알고리즘 기법이 바로 <b>동적 계획법(Dynamic Programming)의 메모이제이션</b>입니다. 결과를 매번 새로 계산하면 속도는 느리지만 메모리는 적게 들고, 결과를 캐싱(미리 저장)해두면 메모리를 많이 쓰는 대신 속도가 빨라지는 상충 관계를 이용합니다.

위의 $O(2^n)$ 짜리 피보나치 함수를, 이미 계산한 값을 배열(메모리)에 저장하는 방식으로 바꾸면 <b>공간을 좀 더 쓰고 시간을 획기적으로 단축</b>할 수 있습니다.

```python
def fibonacci_dp(n, memo={}):
    if n in memo:
        return memo[n]  # 이미 계산한 메모리가 있다면 즉시 반환 (시간 단축)
    if n <= 1:
        return n
    
    # 계산 결과를 memo 공간에 저장하여 중복 연산을 완전히 제거
    memo[n] = fibonacci_dp(n - 1, memo) + fibonacci_dp(n - 2, memo)
    return memo[n]
```
* **시간 복잡도:** $O(n)$ $\rightarrow$ 단 한 번씩만 연산하므로 트리 형태의 중복 호출이 사라져 $O(2^n)$에서 빛의 속도로 단축됩니다.
* **공간 복잡도:** $O(n)$ $\rightarrow$ 값을 기억할 `memo` 사전(Dictionary) 공간과 재귀 스택 공간이 추가 보조 공간으로 소모됩니다.

## 8. 한눈에 보는 선형 자료구조 복잡도 총정리

CS 스터디원들의 요약본 내용을 통합하여, 실제 문제를 풀 때 어떤 자료구조를 선택해야 하는지 결정하기 위한 가이드라인 테이블입니다.

| 자료구조 | 접근 (Access) | 삽입/삭제 (Insertion/Deletion) | 특징 및 비유 |
| :--- | :--- | :--- | :--- |
| **배열 (Array)** | **$O(1)$** | $O(n)$ | 번호가 지정된 아파트 우편함. 연속된 메모리에 위치하여 인덱스를 통한 랜덤 접근이 가능해 탐색이 빠르지만, 중간 삽입/삭제 시 칸을 밀고 당겨야 함. |
| **연결 리스트<br>(Linked List)** | $O(n)$ | **$O(1)$** | 기차 객차 연결 또는 보물찾기 쪽지. 메모리에 흩어져 노드의 포인터 주소로 연결되므로 특정 요소 접근은 느리지만, 중간 삽입/삭제는 연결선만 바꾸면 되어 아주 빠름. |
| **동적 배열 (Vector)** | **$O(1)$** | $O(1)$ (평균) | 칸이 자동으로 확장되는 가변형 서랍장. 공간 부족 시 2배씩 메모리를 늘려 데이터를 복사하는 분할 상환 분석(Amortized Analysis) 덕분에 평균 $O(1)$을 보장함. |
| **스택 (Stack)** | $O(n)$ | **$O(1)$** | 밑이 막힌 프링글스 과자통 구조. 후입선출(LIFO). 데이터 삽입/삭제가 맨 위인 TOP에서만 발생하며 웹 브라우저 뒤로가기, 실행 취소(Ctrl+Z)에 활용됨. |
| **큐 (Queue)** | $O(n)$ | **$O(1)$** | 양쪽이 뚫린 맛집 선착순 줄서기, 터널 구조. 선입선출(FIFO). 삽입은 Rear, 삭제는 Front 양 끝에서 발생하며 프린터 대기열, 은행 번호표 시스템에 활용됨. |
| **해시 테이블<br>(Hash Table)** | **$O(1)$** (평균) | **$O(1)$** (평균) | 마법의 함수 번호표와 사물함 시스템. Key-Value 쌍으로 이루어져 해시 함수를 통해 데이터의 주소 위치를 단번에 즉시 찾아내므로 극도의 효율을 자랑함. |
